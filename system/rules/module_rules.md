# ⚙️ СИСТЕМНОЕ ПРАВИЛО: Ведение модулей (VtM v20)

> 📌 Протокол создания и хранения файлов модулей. Город-нейтрально: `<город>` — slug города; `<хроника>` — slug хроники; `<модуль>` — ASCII-слаг модуля.
> ✅ Сверено со структурой `cities/<город>/`.
> 🔗 Работает в связке с [`chronicle.md`](chronicle.md) и [`diary_rules.md`](diary_rules.md). Операционный чеклист — [`CHECKLIST.md`](CHECKLIST.md) (секция 2).

---

## 📂 СТРУКТУРА — ЕДИНСТВЕННО ВЕРНЫЙ ФОРМАТ

Модули живут внутри своей хроники:

```
cities/<город>/chronicles/<хроника>/modules/
└── <модуль>/                  ← ASCII-слаг (строчными, через _)
    ├── <модуль>.md            ← всегда; главный файл; имя = имя папки
    ├── scenario.md            ← инструмент Мастера до сессии
    ├── sessions.md            ← Фаза B: журнал сессий (если модуль идёт >1 сессии)
    ├── finale.md              ← только если сессия разыграна и финал написан
    ├── npc.md                 ← список всех НПС (каноничных и модульных)
    └── npc/                   ← папка модульных (неканоничных) НПС
        └── <slug>/
            └── <slug>.md      ← упрощённый лист (Шаблон Г из npcs_city.md)
```

**Правила именования папки модуля:**
- ASCII-слаг строчными буквами, через `_` (например: `progulki_po_nocham`, `priglashenie_domoy`, `la_defence`)
- Слаг формируется из названия модуля транслитерацией (как делает веб-интерфейс / `tools/new_npc.js`)

**Главный файл модуля:** `<модуль>.md` (имя = имя папки). Не использовать `module.md` — неинформативно.

## 🔁 Протокол создания модуля

> ⚡ **Веб-воркфлоу — три фазы жизненного цикла.** Модуль ведётся кнопками веб-интерфейса (`web/`); отдельного CLI нет.
>
> **Фаза A — Подготовка (до игры).** Два шага:
> - **«+ Модуль»** (заготовка): папка модуля, главный файл `<модуль>.md` (метаданные + участники + раздел `## 💡 Концепция` с **идеей**) и обновление `## 🔗 Модули` в `chronicle.md`. Поле **«Время»** обязательно. Достаточно одной идеи.
> - **«🪄 Сгенерировать»** (сборка): из `## 💡 Концепция` и участников создаёт `scenario.md`, карточки **новых** локаций (`locations/Другие/<модуль>/…`) и **эпизодических НПС** (`npc/<slug>/<slug>.md`, Шаблон Г) и `npc.md` — всё **с проверкой переиспользования** (существующие локации/персонажи не дублируются, а переиспользуются). Концепция и участники не затираются — можно перегенерировать.
>
> **Фаза B — Ведение во время игры.** Модуль не всегда проходится за одну сессию. Каждую сыгранную сессию фиксируй во вкладке **«🎲 Сессии»** страницы модуля (кнопка «+ Запись сессии») → пишется в `sessions.md`. Это **рабочий журнал Мастера**, а не канон: что сыграно, какие сцены пройдены, текущий статус модуля. Канон фиксируется на Фазе C.
>
> **Фаза C — Закрытие.** Когда модуль доигран: кнопка **«🔒 Закрыть модуль»** на странице модуля запускает Фазу C — генерирует `finale.md` по фактам игры (из `sessions.md`), переносит каноничное событие в `chronicles/<хроника>/events.md` (пересборка агрегата `archive/events.md`) и ставит статус модуля 🟢 Закрыт. Остальное (нити, дневники участников, продвижение модульных НПС, `validate_links`) — вручную; протоколы: [`chronicle.md`](chronicle.md), [`diary_rules.md`](diary_rules.md); чеклист — [`CHECKLIST.md`](CHECKLIST.md) §2. ⚠️ Закрытие **модуля** идёт по правилам модуля (Фаза C), а не по `close_chronicle.js` (то — закрытие **хроники**).
>
> **Удаление.** Кнопка **«🗑 Удалить модуль»** удаляет папку модуля, его блоки событий из `chronicles/<хроника>/events.md` (+ пересборка агрегата), ссылки из дневников участников и модульные (неканоничные) НПС из `npc/`. Каноничных персонажей и их карточки не трогает.

### 🧭 Правила генерации сценария (Фаза A — «Сгенерировать»)

1. **Проверка таймлайна.** Поле **«Время»** обязательно при создании модуля (желательно с годом). Генератор сверяет дату модуля со статусом/ролью/датами в карточках участников и в начале сценария выдаёт раздел `## ⚠️ Проверка таймлайна` с предупреждениями Мастеру: если на эту дату персонаж **не мог участвовать** или его статус был иным — ещё не на должности, в торпоре, ещё не обращён/не прибыл, уже уничтожен. *Пример: модуль «Метро» (декабрь 2009) — Мэриам Дэвис ещё не Шериф (назначена ноябрь 2010); Клер де Кюстин ещё в торпоре.* Если конфликтов нет — пишется «Конфликтов таймлайна не выявлено».
2. **Связи между персонажами.** Генератор учитывает поле `Отношения` карточек участников и вплетает связи (сир/дитя, союзник/враг, долг) в мотивации, конфликты и сцены.
3. **Именованные эпизодические НПС.** Каждый НПС, включая эпизодических антагонистов и второстепенных (охотник, информатор, гуль и т.п.), получает **конкретное имя** и попадает в раздел «НПС» сценария отдельным пунктом «- Имя — роль». Безымянных функциональных персонажей в разделе НПС быть не должно — иначе для них **не создастся карточка** `npc/<slug>/<slug>.md`.
4. **Переиспользование локаций (по имени и по типу).** Локации из раздела «Локации» сценария сверяются с `locations/`: совпадение по имени → переиспользуется. Для **типовых мест** (станция метро, катакомбы, кладбище и т.п.) действует переиспользование **по типу**: нельзя плодить однотипные локации с разными названиями. *Пример: если действие в метро — используется уже существующая станция, а новую станцию создаём только для округа, где её ещё нет.* Карточка создаётся только для действительно нового места (`locations/Другие/<модуль>/…`).

### 🎲 Фаза B — журнал сессий (`sessions.md`)

Модуль может идти несколько сессий. Каждую сыгранную сессию фиксируй во вкладке **«🎲 Сессии»** (кнопка «+ Запись сессии») — добавляется блок в `sessions.md`. Это рабочий журнал, **источник истины при закрытии модуля** (Фаза C): из него собираются записи в хронику и дневники.

Формат записи:

```markdown
## Сессия N — ГГГГ-ММ-ДД

- **Сыграно сцен:** Сцена 1, Сцена 2
- **Статус модуля:** 🟡 В процессе   <!-- или 🟢 Завершён -->

Что произошло за сессию: события, решения котери, последствия.
```

Правила:
- Нумерация сессий — сквозная, дата — дата реальной игры.
- **Сыгранные сцены** выбираются из выпадающего списка (сцены берутся из `scenario.md`); уже отмеченные в прошлых сессиях сцены из списка исключаются. За одну сессию можно отметить **несколько сцен**.
- **«События вне сценария»** — постоянный пункт списка (не исчезает): VtM не всегда идёт по сценарию, игроки могут отступать от него. Отмечай им внесценарные действия котери.
- Любую запись сессии можно **отредактировать** (кнопка ✏️ на карточке) — результаты сыгранной сцены уточняются по факту игры.
- **Статус 🟢 Завершён** — сигнал, что пора переходить к Фазе C (закрытие).
- Журнал **не заменяет** канон: окончательные события идут в `events.md`, дневники — в `journal/` (Фаза C).

Если делаешь вручную — выполни шаги **в строгом порядке**:

0. **Проверить связь с предыдущими хрониками.** Если модуль продолжает другой — найти модули с `finale.md`, выбрать нужный, вынести ссылку на финал в `scenario.md` (раздел «Предпосылки»).
1. **Создать папку:** `cities/<город>/chronicles/<хроника>/modules/<модуль>/` и подпапку `npc/`.
2. **Создать главный файл `<модуль>.md`:** таблица параметров + краткое содержание. До сессии допустима заглушка.
3. **Создать `scenario.md`:** Предпосылки → Локации → НПС → Завязка → Сцены (3–5) → Кульминация → Варианты финала → Крючки → колорит города. Заполняется Мастером до сессии или совместно с Claude по запросу.
4. **Создать `npc.md`:** три секции — ПК / Каноничные НПС (ссылки на `characters/`) / Модульные НПС (ссылки на `npc/`). Новых определить как каноничных или модульных (см. ниже).
5. **Создать карточки новых локаций:** если модуль использует локацию без карточки — создать по правилам [`portret.md`](portret.md):
   - В пределах города → `cities/<город>/locations/district_NN/<район>/<локация>/<локация>.md` (глубина гибкая)
   - Вне города → `cities/<город>/locations/Другие/<модуль>/<локация>/<локация>.md`
   - Карточка: атмосфера, сенсорная палитра, VtM-контекст, 3 крючка, промт + негативный промт. Проверить наличие: `tools/search.ps1 "Название локации"`.
6. **Создать `finale.md`:** только если сессия разыграна и литературный финал написан.
7. **Добавить запись о событии** в `cities/<город>/chronicles/<хроника>/events.md` по протоколу из [`chronicle.md`](chronicle.md) (шаги 1–7).

> 📋 Полный операционный чеклист (дневники участников, `open_threads.md`, валидация ссылок) — [`CHECKLIST.md`](CHECKLIST.md) → секция 2.

## 🆕 Модульные (неканоничные) НПС

НПС, созданный специально для одного модуля, по умолчанию **неканоничный**: хранится в `npc/<slug>/<slug>.md` по **Шаблону Г** из [`npcs_city.md`](npcs_city.md) (упрощённый лист с характеристиками + промт). В `characters/` не добавляется, в `characters_index.md` не вносится.

### Условия продвижения в каноничные

НПС становится каноничным при **выполнении всех трёх условий**:
1. Выжил (не уничтожен, не в торпоре постоянно).
2. Упомянут в `finale.md` как значимый персонаж.
3. Появился минимум в двух разных модулях — **или** пользователь явно запросил канонизацию.

**Протокол продвижения:**
1. Создать полную карточку по Шаблону А/Б/В в `cities/<город>/characters/<линейка>/<slug>/<slug>.md`.
2. Добавить запись в `cities/<город>/archive/characters_index.md`.
3. Обновить `npc.md` в каждом модуле, где фигурирует НПС: заменить ссылку `npc/<slug>/<slug>.md` на `../../../../characters/<линейка>/<slug>/<slug>.md`.
4. Удалить файл из `npc/` (или оставить с пометкой `→ перемещён`).
5. Запустить `tools/validate_links.ps1 -Force`.

## ⚠️ Правила и ограничения

- Папка модуля создаётся **всегда** при добавлении записи в хронику — даже если модуль не разыгрывался как сессия (достаточно заглушки).
- Карточка локации создаётся **при первом появлении** локации в любом модуле. Если локация уже есть в `locations/` — достаточно ссылки. Проверить: `tools/search.ps1 "Название локации"`.
- `finale.md` создаётся **только при необходимости** — не заранее.
- `npc.md` создаётся **всегда для игровых сессий** — перечислять всех НПС модуля. **Исключение:** внесессионные события могут не иметь `npc.md`, если НПС уже перечислены в тексте главного файла.
- Ссылки на дневники: если событие до ноября 2010 → ссылаться на `journal/retrospective.md`, не на месячный файл (`journal/ГГГГ-ММ.md`).
- Если пользователь словесно описывает иную структуру → задать уточняющий вопрос с объяснением последствий (приоритетный принцип — см. [`diary_rules.md`](diary_rules.md)).
