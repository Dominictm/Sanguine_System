# Библиотека → «Факты биографии» (Backgrounds) — дизайн-спека

Дата: 2026-07-09
Аналитик: параллельное исследование — (1) кодовая база (блюпринт Достоинства/
Недостатки), (2) содержимое вики wod.fandom.com по Дополнениям.

## 1. Что уже есть в проекте (источник для копирования)

Полный технический разбор Достоинств/Недостатков — см. отчёт исследования
(воспроизведён ниже кратко). Новый раздел **зеркалит эту же архитектуру
один-в-один**, ничего архитектурно нового не изобретаем.

- **Данные**: `system/library/{merits,flaws}/<категория>.json` — массив
  `{ slug, name, points, description, category }`.
- **Загрузчики**: `web/lib/{merits,flaws}-loader.js` — кэш в памяти по
  категории, `getX(category)` + `getAllX()`.
- **Роуты**: `GET /api/library/merits/:category`, `GET /api/library/merits`
  (плоский список всех категорий) — по образцу.
- **Фронтенд** (`web/public/v20-sheet.js`): кэш `{cat: null, ...}`,
  `_libXCardsHtml(category)` → `.lib-cards`/`.lib-card` (уже общие классы,
  не специфичные для достоинств), `_v20OpenXModal(slug, category)` →
  общая модалка `_v20EnsureLibModal()` (уже одна на все 4 текущих раздела —
  дисциплины/психика/достоинства/недостатки).
- **HTML**: вкладка в `#page-library .tab-bar`, суб-табы категорий
  (`.merits-subtab-bar`/`-btn`, уже общий CSS-класс с недостатками).
- **Важное отличие для Фактов биографии**: у достоинств/недостатков есть
  смысловое деление «тип» (Достоинство/Недостаток) с очками; у Дополнений
  такого деления нет — только положительная инвестиция 1–5 точек. Значит
  схема данных даже проще (не нужно поле типа), но **сама сущность (карточка
  с очками-точками) переиспользуется 1-в-1** из уже готового `.lib-card-points`
  + `.lib-dot` (тот самый компонент, который только что перекрасили в
  красные точки с золотым бордером).

## 2. Категории (утверждено с пользователем)

Вместо полного вики-разбиения по сеттингам (вампиры/маги/гару/подменыши/
демоны/призраки/катаяны — ~90+ записей, из которых большинство неприменимо
к текущим линейкам проекта) — **5 категорий**, отобранных под линейки WoD,
которые проект реально поддерживает (`vampires`, `fairies`, `mortals`,
`werewolves`, `mages`, `hunters` — см. CLAUDE.md):

| Категория (slug) | Русское имя вкладки | Источник (раздел вики) | Примерно записей |
|---|---|---|---|
| `general` | Общие | «Общие» (3.1) | ~15 |
| `vampire` | Вампиры | «Вампиры» (3.2) + шабашские Дополнения (раздел 2, помечаются как «только Шабаш» в описании, не отдельной категорией) | ~15 |
| `ghoul` | Гули и ревенанты | «Гули, ревенанты и дампиры» (3.3) | 4 (см. п.3 — фильтрация) |
| `mage` | Маги | «Маги» (3.6) | ~10 |
| `changeling` | Подменыши | «Подменыши» (3.9) | ~7 |

Итого ориентировочно **~51 запись** на первую версию — сопоставимо по
объёму с существующими Достоинствами (62 записи на 4 категории).

**Важно (найдено при анализе вики):** в разделе «Гули, ревенанты и дампиры»
сам редактор вики пометил блок из 11 пунктов (Доброжелательный Домитор,
Наследный гуль, Кровная связь, Иммунитет к Недостаткам Крови, Смертельный
Удар, Сексуальность, Шпион, Бледная аура, Истинный цыган, Кинфолк, Чутьё на
вампиров) припиской **«А это ДОСТОИНСТВА, а не Дополнения»** — это
Merits, ошибочно попавшие в список Backgrounds на странице-источнике.
**Эти 11 пунктов НЕ импортируем** в Факты биографии — категория `ghoul`
реально содержит только 4 подлинных дополнения (Двойная жизнь, Старейшина
семьи, Мажордом, Домитор).

## 3. Источник полного текста

Обзорная страница-список (`.../Дополнения#Признаки`) содержит только
**одну строку сути на дополнение**, без по-точечной механики (что даёт 1
очко vs 5 очков и т.д.) — сопоставимо с оглавлением, а не карточкой. Полный
текст лежит на **отдельной вики-странице на каждое дополнение**
(`wod.fandom.com/ru/wiki/<Название>`, например `.../Ресурсы`, `.../Слуги`,
`.../Домен`).

**Утверждено:** дозапросить ~46 отдельных страниц (51 минус то, что может
не иметь отдельной страницы — уточнится на месте) через тот же метод,
которым уже успешно обошли Cloudflare-блок на прямой WebFetch/curl —
Fandom API `action=parse&page=<Название>&prop=wikitext`. Это отдельная,
объёмная задача — выносится в **Задачу 1** плана реализации (см. ниже),
не блокирует остальную архитектуру.

## 4. Структура данных (одна запись)

```json
{
  "slug": "resursy",
  "name": "Ресурсы",
  "points": null,
  "description": "…(литературное описание, как у Достоинств)…",
  "system": "●○○○○ — Вы едва сводите концы с концами.\n●●○○○ — …\n…",
  "category": "general"
}
```

Отличие от Достоинств: поле **`points` не фиксировано у самого Дополнения**
(это не «стоимость 2 очка», а «шкала 1–5, эффект зависит от того, сколько
точек вложил игрок») — значит на карточке показываем не `'●'.repeat(m.points)`
жёстко, а **шкалу 1–5 точек как справочную иллюстрацию** (аналогично тому,
как дисциплины показывают уровни 1–10 через `_libPowerHtml`/`_libLevelDots` —
компонент для этого уже есть, просто переиспользуем на диапазон 1–5).
Поле `system` — по-точечный текст (что даёт каждый уровень), парсится из
вики-таблицы уровней на отдельной странице дополнения; `description` —
литературное описание (вводный абзац).

**Указание пользователю на будущее (не блокирует эту фичу):** т.к. у
Дополнений нет фиксированной «стоимости», в отличие от Достоинств/
Недостатков, карточка/модалка дополнения физически ближе по форме к
карточке **дисциплины** (уровни 1–5, литературное + системное описание),
чем к карточке достоинства (фиксированные точки). Рекомендация ниже
(п. 6) — использовать `_libPowerHtml`-подобный рендер уровней, а не
`_libMeritDetailHtml`-подобный.

## 5. Бэкенд

- `system/library/backgrounds/{general,vampire,ghoul,mage,changeling}.json`
  — 5 файлов, схема см. п.4.
- `web/lib/backgrounds-loader.js` — точная копия `merits-loader.js`
  (кэш по категории, `getBackgrounds(category)`, `getAllBackgrounds()`,
  список категорий `['general','vampire','ghoul','mage','changeling']`).
- `web/routes/library.js` — добавить:
  - `GET /api/library/backgrounds/:category`
  - `GET /api/library/backgrounds` (плоский список всех категорий)

## 6. Фронтенд

- HTML (`index.html`): пятая вкладка `data-tab="lib-backgrounds"` в
  `#page-library .tab-bar` (унифицированный стиль табов — как уже сделано
  для остальных четырёх), суб-таб-бар `.backgrounds-subtab-bar`/`-btn`
  (5 кнопок по категориям, тот же общий CSS, что и
  `.merits-subtab-bar`/`.flaws-subtab-bar`).
- `web/public/v20-sheet.js`:
  - `_backgroundsCache = { general: null, vampire: null, ghoul: null, mage: null, changeling: null }`
  - `_libBackgroundCardsHtml(category)` — карточка: имя + шкала уровней
    1–5 (см. п.4), не фиксированные очки.
  - `_libBackgroundDetailHtml(b)` — литературное описание + по-точечная
    система (как у дисциплины: `_libPowerHtml`-подобный рендер, не
    `_libMeritDetailHtml`-подобный).
  - `_v20OpenBackgroundModal(slug, category)` → та же общая модалка
    `_v20EnsureLibModal()` (пятый потребитель, back-кнопка = `data-modal-close`,
    как у достоинств/недостатков — нет отдельного списка-внутри-модалки).
  - `loadBackgroundsLibrary(category)` — по образцу `loadMeritsLibrary`.
- `web/public/scripts.js`: делегированный обработчик суб-табов
  `.backgrounds-subtab-btn` — по образцу существующих (строки ~613-637).
- CSS: переиспользуется `.lib-cards`/`.lib-card`/`.lib-card-name` без
  изменений; суб-таб-бар — добавить `.backgrounds-subtab-bar`/`-btn` в
  существующий общий селектор `.merits-subtab-bar, .flaws-subtab-bar { … }`
  (расширить comma-selector, не дублировать блок).

## 7. Итоговая оценка объёма

Технически — прямое повторение уже проверенного паттерна (нулевой
архитектурный риск, всё уже отработано на 4 разделах библиотеки).
Основной объём работы — не код, а **подготовка данных**: дозапрос и
структурирование ~46 отдельных вики-страниц в 5 JSON-файлов с разбором
по-точечного текста. Рекомендуемый порядок задач в плане реализации:

1. Дозапрос вики-страниц + сборка 5 JSON-файлов (данные) — самая долгая
   задача, полностью независима от кода.
2. Backend: loader + роуты (копия merits, ~30 минут).
3. Frontend: карточки + модалка + суб-табы (копия merits/flaws,
   с изменённым рендером уровней вместо точек — см. п.4/6).
4. HTML/CSS: пятая вкладка, суб-таб-бар.
5. Тесты: по образцу существующих тестов для `/api/library/merits`.
6. Визуальная проверка headless Chrome — все 5 категорий, карточка → модалка.

Готов перейти к написанию пошагового implementation plan (`docs/superpowers/plans/`)
после вашего подтверждения этой спеки — или сразу приступить к Задаче 1
(дозапрос вики-контента), если архитектура одобрена.
