## Фаза J — дизайн страницы города (S)

**Найдена причина бага** (не просто «полировка» — реальный дефект):
`#page-city` использует разметку и классы, спроектированные для **модалки**
персонажа (`.cdet-sticky-header`, `.cdet-name`, `.cdet-panels{overflow-y:auto}`),
где высота задаётся `.modal-box` (`display:flex; flex-direction:column`,
`max-height`). На **странице** такого ограничителя нет — базовый `.page`
сам скроллится (`height:100vh; overflow-y:auto`), а вложенный
`.cdet-panels` тоже пытается скроллиться, но без заданной высоты просто
растягивается. Результат — тот самый «боковой скролер на всю страницу»,
плюс декоративная шапка (клановый градиент, зерно текстуры, огромный
`--fs-display-3` заголовок), рассчитанная на карточку персонажа, смотрится
чрезмерно на служебной странице.

**Решение — взять готовый паттерн `#page-module`** (пример пользователя
«Встреча в парке» — это буквально ссылка на этот паттерн):
`web/public/index.html:242`, `styles.css:7208-7218` — `padding:0;
overflow:hidden`, `.active{display:flex;flex-direction:column}`, тонкий
`.modp-header` (не растягивается) + `.modp-body{flex:1;overflow:hidden}` →
`.modp-panel{height:100%;overflow-y:auto}`.

**Объём:**
- `#page-city` получает те же CSS-правила, что `#page-module` (padding 0,
  flex-колонка).
- Новая тонкая шапка (по образцу `.modp-header`/`.modp-title`, не
  `.cdet-sticky-header`): название города, год, действия («✏
  Редактировать», «🗑 Удалить», «Сделать активным») — без кланового
  градиента и зерна (это городская страница, не карточка персонажа).
- `_renderCityView`/`_renderCityEdit` (`city.js`) переносят контент секций
  в один скроллящийся контейнер по образцу `.modp-panel` вместо
  `.cdet-panels`.
- Приёмка: CDP-замер `document.body.scrollHeight` vs `innerHeight` при
  открытой странице — не должно быть двойного скролла; визуальный размер
  шапки сопоставим с `.modp-header`.

## Фаза G — фильтр по линейке WoD на графе связей (S)

Страница «🕸 Связи» (`#page-graph`, `web/public/index.html:172`) сейчас не
имеет фильтров вообще — только легенда типов связей, зум и сброс
(`graph.js:113` `buildLegend`). Персонажи графа уже несут `lineage`
(`vampire`/`fairy`/`mortal`/…) — то же поле, что и `filter-lineage` на
странице «Персонажи» (`index.html:135`).

**Объём:**
- Панель чекбоксов/мультиселект линеек в `#graph-toolbar` (тот же список
  опций, что `#filter-lineage` на Персонажах — переиспользовать
  `LINEAGE_LABELS`/`LINEAGE_ICONS` из `scripts.js`, не дублировать).
- Фильтрация на клиенте: скрыть узлы не выбранных линеек (`nodeG.filter`,
  `link` тоже скрывать, если оба конца невидимы) — без похода на сервер,
  `/api/graph` не меняется.
- Состояние фильтра не персистится между сессиями (простая версия);
  сброс — существующая кнопка «↺ Сброс» снимает и фильтр линеек.
- Приёмка: CDP — снять галочку «Феи» → узлы-феи и их связи скрыты, счётчик
  видимых узлов меньше общего.

## Фаза H — вкладка «Эпизодические персонажи» (M)

**Текущее состояние:** 6 линеек-папок (`vampires/fairies/mortals/
werewolves/mages/hunters`), плюс модульные НПС живут *внутри* модуля
(`chronicles/<хроника>/modules/<модуль>/npc/`) и промоутятся в постоянные
персонажи через `_checkNpcPromotion`/`_npcSheetPaths` (`web/lib/db.js`,
протокол — `system/rules/npcs_city.md`). Отдельного места для «одноразовых»
персонажей города (не привязанных к модулю, не претендующих на постоянство)
нет.

**Решение:** седьмая псевдо-линейка `episodic` — папка
`characters/episodic/<slug>/` того же формата карточки, что остальные
(проходит `validate_cards.js` без исключений), но:
- не участвует в графе связей по умолчанию (слишком шумно) — но не
  запрещать, просто не превращать в дефолт;
- не имеет V20-листа по умолчанию (эпизодический — обычно без полного
  листа), но кнопка «Полноценный лист» остаётся доступна, если понадобится.

**Объём:**
- `LINEAGE_MAP` (`web/lib/db.js:78`) — добавить `episodic: 'episodic'`
  (или человеческое имя-линейка, решить в спеке).
- Новая вкладка на странице «Персонажи»: либо суб-таб внутри существующей
  страницы (переключатель «Постоянные / Эпизодические» рядом с фильтром
  линейки), либо отдельный пункт навигации — решить в спеке (первое
  дешевле и логичнее, раз это тот же список персонажей).
- Создание/удаление — существующие `POST /api/characters` /
  soft-delete-DELETE, просто с `lineage: 'episodic'`; форма создания
  переиспользуется, только добавляется этот вариант в `lineage-picker`
  (`index.html:1189`).
- Тесты: round-trip создания/чтения/удаления эпизодического персонажа;
  `validate_cards.js` не падает на новой папке.
- Приёмка: CDP — создание, отображение в своей вкладке, удаление.

## Фаза K — структурная форма «Ключевые локации» (L)

**Ключевая находка:** локации в проекте — **не текст**, а полноценные
карточки с богатой структурой (проверено на `cities/paris/locations/.../
istoricheskiy_ostrov.md`): округ, район, адрес, зона Маскарада (🟢/🔴),
контроль фракций, статус, угрозы, сенсорная палитра, крючки. Ровно то, что
просит пример пользователя («Район №14 Эль Мури, станция метро "Жалю",
отметки Маскарада/опасности, статус — место сбора носферату») — это
**один в один существующая карточка локации**, только через
`POST /api/locations` (`web/routes/locations.js:270`, поля `name`,
`district`, опционально AI-генерация) + `PUT /:slug/fields`.

Свободное поле «Ключевые локации» в `city.md` (`CITY_SECTIONS.locations`)
сейчас дублирует эту систему текстом-заглушкой («Небоскрёб в центре,
Элизиум: Опера») — вместо ссылок на реальные карточки.

**Рекомендация (не плодить две системы):** заменить textarea секции
«Ключевые локации» в форме создания/редактирования города на **встроенный
мини-конструктор локаций**, использующий существующий API:
- В форме города: список уже добавленных «ключевых локаций» (карточки
  локаций с флагом «ключевая» или просто локации без района → «Другие»,
  решить в спеке) + кнопка «+ Добавить ключевую локацию», открывающая
  компактную под-форму: Название, Район (datalist из уже введённых в поле
  «Районы»), краткий статус, опционально «🪄 Сгенерировать».
- Каждая «ключевая локация» реально создаёт карточку локации через
  существующий `POST /api/locations` — а не строку текста в `city.md`.
  Секция `city.md` «Ключевые локации» превращается в авто-генерируемый
  список ссылок на эти карточки (по образцу того, как `archive/events.md`
  собирается из хроник `tools/build_city_events.js`) **или** остаётся
  ручной пометкой «эти локации — ключевые» (решить в спеке — второе проще
  и меньше рискует сломать существующий рабочий процесс локаций).
- Совместимость: у городов, где «Ключевые локации» уже текст (Париж,
  Балмонт) — текст остаётся читаемым как есть (миграция не нужна, секция
  просто получает новый способ пополнения рядом со старым текстом).

**Расширение на будущее** (пользователь: «в дальнейшем будем добавлять
пункты») — эта фаза задаёт **паттерн под-конструктора** внутри формы
города, применимый и к другим секциям D1 (напр. «Районы» уже могли бы
получить аналогичный конструктор вместо textarea, «Значимые места» —
аналогично локациям). Не делаем это сейчас для всех секций — только для
«Ключевых локаций» как пилот, остальные — по мере запроса.

**Приёмка:** CDP — добавление «Район №14 Эль Мури, станция Жалю» через
под-форму создаёт настоящую карточку локации в `locations/district_14/…`,
видимую на странице «Локации» и доступную AI-генерации сценариев
(существующий каталог локаций в промтах, `web/routes/modules/fill.js`).

## Фаза I — авторские элементы библиотеки (L, самая крупная)

**Текущее состояние — два разных формата хранения**, оба read-only (нет
веб-формы создания, только серверные загрузчики):
- **Дисциплины/психика** — по одному `.md`-файлу на элемент
  (`system/library/disciplines/*.md`, `psychics/*.md`), формат
  задокументирован в `system/library/disciplines/README.md`: заголовок +
  метаданные + уровни `## Уровень N` с «Литературное описание»/«Система».
- **Достоинства/недостатки/бэкграунды** — JSON-массивы по категориям
  (`system/library/merits/{physical,social,mental,supernatural}.json`,
  аналогично `flaws/`, `backgrounds/`).

**«Факты биографии»** — упомянуты пользователем, но такой библиотеки в
проекте пока нет вообще (нет ни файла, ни загрузчика, ни API). Это не
доработка существующего — это **новая категория библиотеки**, требует
отдельного решения в спеке: формат хранения (вероятно, JSON-массив коротких
шаблонов-затравок, ближе к merits/flaws, чем к дисциплинам), где
используется (промт биографии персонажа — `characters.js` generate-
biography уже существует, подмешать оттуда).

**Объём (двухтрековая реализация — форматы разные):**
1. **Трек MD** (дисциплины, психика): веб-форма → собирает поля по
   контракту README → пишет новый `.md` в `system/library/<cat>/<slug>.md`
   → инвалидирует серверный кэш (`web/lib/disciplines.js`/`psychics.js`
   кэшируют в памяти — нужен сброс кэша после записи, не только при
   рестарте).
2. **Трек JSON** (достоинства/недостатки/бэкграунды, + новая «Факты
   биографии»): веб-форма → добавляет объект в массив нужной категории
   JSON-файла → сброс кэша `merits-loader.js`/`flaws-loader.js`/
   `backgrounds-loader.js` (тот же паттерн кэша).
3. Общий UI: на странице «Библиотека» — кнопка «+ Добавить» в каждой
   вкладке (дисциплины/психика/достоинства/недостатки/бэкграунды/факты
   биографии), открывающая форму под формат конкретной категории.
4. Удаление/редактирование авторских записей — не только добавление
   (иначе опечатку не исправить) — CRUD, не только Create.
5. Разграничение канон/авторское: пометка в UI (значок), чтобы не путать
   с официальным V20-контентом — по аналогии с «🏙️» у городских событий в
   `timeline.md` (🏙️ = установлено в проекте, не канон).

**Открытые вопросы для спеки:** нужна ли валидация уникальности
названия/слага внутри категории; нужен ли экспорт/бэкап авторских
добавлений отдельно от кода (они физически лягут в `system/library/`,
которая уже под git — конфликтов с релизной сборкой не будет, `system/`
не исключается `release-exclude.json`, значит авторские дисциплины
пользователя **уедут в релизную сборку build_release.js** — стоит явно
решить, это желаемо или нет, до реализации).

**Приёмка:** CDP — создание кастомной дисциплины появляется в
`GET /api/library/disciplines` без рестарта сервера; то же для достоинства;
удаление работает; линтер/тесты не спотыкаются о новые файлы.

## Сквозные правила

Как обычно в проекте: каждая фаза — TDD (тест → фикс → зелёные), CDP-
проверка, уборка тестовых данных, коммит на фазу. Фаза I меняет формат
только *новых* файлов (не трогает существующие канонические) — миграция
не требуется, но кэш-инвалидация обязательна.
