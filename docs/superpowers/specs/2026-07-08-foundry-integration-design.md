> цель: собрать ground truth и предложить архитектуру для последующего решения с пользователем.

> вторичную установку Foundry и не нашёл ни исходников системы, ни реального VTM-мира. В этой

> строки `meritsFlaws` с канонической библиотекой `system/library/{merits,flaws}/*.json` (та же,
> что наполняет раздел «Библиотека» веб-интерфейса); совпавшие строки экспортируются как

### 1.3 Официальный источник схемы — `template.json` найден и совпадает по версии

Репозиторий **[github.com/JohanFalt/Foundry_WoD20](https://github.com/JohanFalt/Foundry_WoD20)**
(`system.json`: `"id": "worldofdarkness"`, `"version": "7.1.5"`) — **точно та же версия**, что указана
в `parizh-20/world.json` (`systemVersion: "7.1.5"`). Значит установленная у пользователя система —
это (предположительно немодифицированная) копия этого репозитория, и его `template.json`/`wod.js`
можно использовать как достоверный источник схемы без риска расхождения версий.

Композиция Actor-типа `Vampire` (`template.json`):

```json
"Vampire": { "templates": ["base", "ability", "mortal", "vampire"] }
```

То есть `Actor.system` вампира — это слияние 4 партиалов. Ключевые поля по партиалам (сокращено,
дефолты и служебные ключи `label/isvisible/isfavorited/speciality` опущены для краткости):

**`base`** — `traits.health.totalhealthlevels.{value,max}` (7/7 по умолчанию — количество уровней
здоровья, не сами уровни повреждений), `name`, `player`, `nature`, `demeanor`, `concept`,
`appearance`, `background`, `notes`, `gear`, `derangement`, `variant`.

**`ability`** — `abilities.<key>` — **плоский словарь фиксированных ~43 ключей** (не массив
произвольных имён!), каждый вида `{type: "talent"|"skill"|"knowledge", value, max: 5, speciality}`.
Примеры ключей: `alertness`, `athletics`, `brawl`, `dodge`, `firearms`, `melee`, `stealth`,
`academics`, `investigation`, `occult`, `medicine`, `technology` и т.д.

**`mortal`** — `attributes.<key>` (11 ключей, не 9 — включает `composure` и `resolve` в дополнение
к классическим физическим/социальным/ментальным): `strength, dexterity, stamina, charisma,
manipulation, appearance, composure, perception, intelligence, wits, resolve`; `health.{damage,
bruised, hurt, injured, wounded, mauled, crippled, incapacitated}` (классическая точечная шкала
уровней повреждений WoD); `advantages.willpower.{permanent,temporary,max,roll}`;
`advantages.virtues.{conscience,selfcontrol,courage}.{permanent,temporary,max,roll}`;
`advantages.bloodpool.{temporary,max,perturn}` (и другие `advantages.*`, специфичные другим
игровым линиям — rage/gnosis/glamour/faith и т.п., у вампира не используются).

**`vampire`** — `custom.{sect,clan}`, `sect`, `clan`, `bloodline`, `weakness`, `generation`
(число, дефолт 13), `generationmod`, `sire`, `conditions.isfrenzy`, плюс набор
`advantages.{yin,yang,yinchi,yangchi,dharma,hun,po,demonchi}` — эти поля относятся к другим
игровым линиям (Kindred of the East/Wraith-подобная механика), для стандартного V20-вампира
**не используются и должны игнорироваться маппером**.

**Дисциплины/Достоинства/Недостатки/Фон — НЕ поля `Actor.system`, а embedded Item-документы.**
Подтверждено дважды: (а) в `template.json` у типа `Vampire` нет поля `disciplines`/`merits`/
`backgrounds`; (б) в `wod.js:483` система сама читает данные через `actor.items.filter(item => ...)`
— стандартный Foundry API встроенной коллекции `Actor#items`. Схема самого Item (`type: "Power"`
для дисциплин, `type: "Feature"` для достоинств/недостатков/фона) — партиалы `settings + base +
feature (+ power)`:

```json
// Item.templates.feature
{ "type": "", "level": "0", "value": 0, "max": 5 }
// Item.templates.power (добавляется поверх feature для type: "Power")
{ "category": "", "dice1": "", "dice2": "", "difficulty": 0, "isrollable": false,
  "spendwillpower": false, "spendblood": false, ... }
```

`system.type` — дискриминатор вида `"wod.types.disciplinepower"` / `"wod.types.merit"` /
`"wod.types.flaw"` (подтверждено также в `compile.js` модуля `biblioteka-master`, где недостатки
компилируются именно с `system.type: "wod.types.flaw"`). `system.level`/`.value`/`.max` — точки
(1–5 для дисциплины, 1–7 для достоинства/недостатка).

### 1.6 LevelDB — эксклюзивная блокировка подтверждена повторно, сейчас база занята реально

Формат хранения (LevelDB/`classic-level`) и факт эксклюзивного `LOCK` подтверждены в ревизии 1
экспериментом на трёх хендлах в одном процессе. В этой ревизии подтверждено **дополнительно**:
сервер Foundry в `F:\Prodgect\...` **прямо сейчас запущен** (`netstat`: `LISTENING 0.0.0.0:30000`,
PID 2800, есть `ESTABLISHED`-соединение — кто-то или что-то держит мир открытым). Попытка открыть
`data/actors`/`data/scenes`/паки этого мира через `classic-level` сейчас **гарантированно
провалится** с `LOCK`-ошибкой — не пробовалось намеренно, чтобы не создавать риск для реальной
активной сессии.

**Более безопасный способ получить образец реального Actor-документа без риска для LevelDB** —
штатная функция Foundry «Export Data» (доступна из контекстного меню документа в UI: правый клик
по актёру → Export Data → скачивается чистый JSON). Она не требует прямого доступа к базе, работает
через официальный документный API Foundry и **безопасна, даже пока мир открыт**. Рекомендуется как
первый практический шаг (см. Фазу 0 в разделе 5) вместо попытки читать LevelDB вручную.

### 1.7 Фаза 0 выполнена — разбор реального Export Data (ревизия 3)

Пользователь выгрузил через штатный Foundry «Export Data» реального персонажа Actor
(`_stats.compendiumSource: "Compendium.world.vazhnye-nps.Actor.TZPpgGTXEzCaBwpf"` — то есть это
подтверждённо содержимое компендиума `vazhnye-nps`, а не тестовый Actor). Разбор файла даёт
следующее, окончательно закрывающее пробелы раздела 2 ревизии 2:

**Путь/Человечность — не отдельные поля, а один объект `advantages.path`:**
```json
"path": {
  "permanent": 4, "value": 0, "max": 10, "roll": 4,
  "bearing": 0, "bearingtext": "", "custom": "",
  "label": "wod.advantages.path.humanity"
}
```
`label` — i18n-ключ выбранного Пути (здесь `wod.advantages.path.humanity` = Человечность);
`permanent`/`max` — сам показатель (аналог Sanguine); `custom` — свободное поле для нестандартного
Пути. **Подтверждено**, гипотеза снята.

**Кастомные способности вне фиксированного словаря — отдельный embedded Item, не поле `abilities`:**
у персонажа есть способности вроде «Фокусы», «Стрельба из лука», «Верховая езда», «Китайская
мудрость» — каждая как отдельный `Item` (`type: "Trait"`, `system.type:
"wod.types.{talent|skill|knowledge}secondability"`), а не запись в фиксированном словаре
`abilities.*`. Это даёт чистое решение риска 2 из раздела 6 (ревизия 2): при экспорте кастомных
способностей Sanguine → Foundry нужно создавать embedded `Item` типа `Trait`, а не пытаться
писать в `abilities.<key>`.

**Дисциплины — двухуровневая структура embedded Item, подтверждена на реальных данных:**
родительский `Item` (`type: "Power"`, `system.type: "wod.types.discipline"`, `value` = рейтинг
дисциплины, например 6 для «Затемнения») + дочерние `Item` того же типа `Power`
(`system.type: "wod.types.disciplinepower"`, `system.parentid` = `_id` родителя) — по одному на
каждую известную способность дисциплины на нужном уровне. Подтверждает гипотезу из 1.3/раздела 2
буквально: `system.parentid` — реальный механизм связи, не предположение.

**Клан/секта — значения хранятся как i18n-ключи, не свободный текст:**
`"clan": "wod.bio.vampire.malkavian"`, `"sect": "wod.bio.vampire.camarilla"` — при экспорте из
Sanguine (где клан/секта — русский текст) маппер должен переводить название в соответствующий
`wod.bio.vampire.<key>` (обратное — из RU-перевода `ru-ru` модуля, раздел 1.4, для примера
`Малкавиан` → `malkavian`), а не писать текст напрямую в поле.

**`Actor.system` содержит больше, чем в `template.json` из партиалов** — сверх атрибутов/
способностей/здоровья и т.п. реальный документ включает `settings` (era, variant, dicesetting,
флаги `has*` — какие механики активны у конкретного листа, напр. `haswillpower`, `haspath`,
`hasbloodpool`, `hasvirtue`: true, `hasrage`/`hasgnosis`/`hasglamour`: false для вампира), а также
производные/вычисляемые поля верхнего уровня (`soak`, `initiative`, `movement`) — которые, судя по
всему, пересчитываются самой системой и не должны экспортироваться Sanguine напрямую (риск
конфликта при обратном импорте — см. обновлённый раздел 6).

**Побочная находка — искажение кодировки при экспорте/просмотре файла.** Полученный от
пользователя JSON отображается с явным mojibake кириллицы (например, имя персонажа и все RU-тексты
описаний). Причина не установлена в рамках этого документа (искажение могло произойти на любом
шаге: сам Export Data Foundry, текстовый редактор пользователя при просмотре, или копирование в
диалог) — это отдельный риск, который нужно закрыть explicit UTF-8-хендлингом в будущих `foundry-
export.js`/`foundry-import.js` (Фаза 1), и перепроверить на файле, прочитанном напрямую с диска, а
не скопированным через промежуточный шаг.

## 3. Архитектурные варианты (обновлено)

Три варианта из ревизии 1 остаются актуальными, но их взаимное соотношение риска изменилось —
самый дорогой риск варианта (a) («писать в LevelDB вслепую по неизвестной схеме») теперь снят.

### (a) CLI-скрипты по образцу `compile.js`/`scrape.js` — batch sync напрямую в LevelDB

- **Требование не изменилось**: Foundry должен быть закрыт на время работы скрипта (эксклюзивный
  LOCK, подтверждено дважды, включая только что — сервер реально держит мир открытым).
- **Риск схемы теперь снят** — `template.json` даёт подтверждённую схему `Actor.system`, а
  `wod.js` подтверждает механизм `actor.items` для embedded-документов. Остаётся техническая
  задача выяснить точный **физический** формат хранения embedded items внутри Actor-документа в
  LevelDB (сериализуются ли они как часть JSON самого Actor'а единым blob'ом, что стандартно для
  Foundry v10+, или отдельными записями со ссылкой на `parent`) — раньше это было «неизвестно
  полностью», теперь это «наиболее вероятная гипотеза с понятным способом проверить» (прочитать
  один экспортированный Actor JSON и посмотреть на структуру `.items`).
- Плюсы/минусы по сравнению с ревизией 1 не изменились: не нужен Foundry-модуль, но не realtime и
  требует закрытия Foundry на время синхронизации — неудобно, если пользователь ведёт сессию с
  игроками в это время.

### (b) Foundry-модуль с REST/WS API внутри запущенного Foundry

Не изменилось относительно ревизии 1 — по-прежнему избыточно по стоимости разработки для
однопользовательского локального инструмента, но теперь стоит отметить: в мире уже установлен
`socketlib` (см. список модулей `parizh-20`, 1.1) — модуль-библиотека для упрощённого
межклиентского общения через сокеты Foundry, которым мог бы воспользоваться будущий
интеграционный модуль, если вариант (b) когда-нибудь станет актуальным. Не меняет рекомендацию.

### (c) Промежуточный JSON-экспорт/импорт через кнопки в Sanguine web UI + штатный Foundry Export/Import

Не изменилось по сути, но теперь **усилено находкой 1.6**: штатный Export Data — не только
безопасная замена прямому чтению LevelDB для разработки (Фаза 0), но и **естественный
транспортный формат для самого варианта (c)** — тот же механизм, который человек будет использовать
руками при каждом переносе персонажа.

### Рекомендация (не изменилась, но обоснование усилилось)

**Начать с (c), затем при необходимости эволюционировать к (a).**

Обоснование дополнено фактами этой ревизии: мир `parizh-20` реально используется для живых игровых
сессий (`playtime: 293936` секунд ≈ 82 часа, `lastPlayed` — сегодня) — то есть Foundry-сессии
происходят регулярно и, вероятно, с игроками, для которых закрытие Foundry ради batch-синхронизации
неприемлемо в моменте. Вариант (c) не требует останавливать сессию вообще — пользователь
экспортирует/импортирует персонажа между сессиями, когда удобно. Вариант (a) остаётся следующим
шагом, если ручной поток станет обременительным — и теперь для него нет блокера по незнанию схемы,
только организационное требование закрывать Foundry на время прогона.

пользователя, раздел 6).

- Обязательная проверка LOCK перед стартом с понятной ошибкой «закройте Foundry» (пользователь

  организационных препятствий для перехода к этой фазе нет, если ручной поток окажется тесным).
- Актуально, только если ручной поток (Фазы 1–2) окажется слишком частым/неудобным.

### Фаза 4 (далёкий горизонт, опционально) — вариант (b): живой Foundry-модуль
- Рассматривать, только если частота синхронизации потребует realtime без остановки сессии.
  В мире уже есть `socketlib` — потенциальная база для реализации.

## 6. Решения по рискам и открытым вопросам (ревизия 4 — обсуждено с пользователем)

Все пункты ревизии 3 обсуждены; ниже — принятые решения (не гипотезы), они фиксируют объём и
подход Фазы 1.

1. **Персонажи в Foundry и Sanguine заведены независимо** (не одна и та же карточка, отражённая в
   двух местах) — подтверждено пользователем. Значит Фаза 1 не может начинаться с «создать с нуля»
   и не может слепо доверять одной стороне — первый практический шаг это **сверка по имени**:
   для каждого персонажа, у которого есть карточка в `cities/paris/characters/vampires/` И Actor в
   Foundry-компендиумах (`personazhi`/`igroki`/`nps`/`vazhnye-nps`), сравнить оба набора данных и
   решить вручную (при первом переносе), какая сторона — источник истины на старте. Дальше
   работает обычный экспорт/импорт по требованию (раздел 4).
2. **`composure`/`resolve`** — добавляются в Sanguine как два новых поля в V20-листе (не
   игнорируются). Требует правки `system/rules/character_sheet_v20.md` (схема листа) и
   соответствующих мест в `web/` (парсер/рендер листа), если и когда Фаза 1 дойдёт до кода — вне
   рамок этого документа, но важно зафиксировать здесь, чтобы это не потерялось при написании
   плана.
3. ~~Фиксированный словарь способностей Foundry vs свободный текст Sanguine~~ **Закрыто**:
   кастомные способности — отдельный embedded `Item` типа `Trait` (раздел 1.7), не поле словаря.
4. ~~Путь Просвещения/Человечность~~ **Закрыто**: единый объект `advantages.path` с i18n-`label`
   (раздел 1.7).
5. **`system.type` для Item-Фона (Backgrounds)** — не проверен на реальных данных (у просмотренного
   персонажа фон пуст). Не блокирует Фазу 1 — проверить при первом тестовом экспорте персонажа с
   непустым Фоном; до этого маппер фона можно оставить недоделанным/помеченным TODO.
6. **`settings` и производные поля (`soak`, `initiative`, `movement`) — маппер их не трогает.**
   Решение: заполнить `system.settings` минимальным набором флагов для вампира
   (`haswillpower`, `haspath`, `hasbloodpool`, `hasvirtue`: `true`; остальные `has*`: `false`) и
   не писать `soak`/`initiative`/`movement` вообще — Foundry пересчитывает их сам при открытии
   листа импортированного Actor'а.
7. **Кодировка UTF-8** — экспортированный пользователем файл дошёл с mojibake кириллицы (раздел
   1.7). Требует явной проверки в Фазе 1: `web/lib/foundry-export.js`/`foundry-import.js` должны
   читать/писать файлы с явной UTF-8 кодировкой и быть протестированы на реальном русскоязычном
   экспорте до заявления готовности.
8. **Форк `Dominictm/Foundry_WoD20_VtM`** признан неактуальным пользователем — можно почистить
   упоминание в README `biblioteka-master` и путь деплоя `FOUNDRY_DIR` в `compile.js` отдельной
   задачей вне рамок текущей интеграции (не блокирует Фазу 1).
9. **Архитектура для Фазы 1 — вариант (c)**, несмотря на то что закрывать Foundry между сессиями
   оказалось приемлемым (открывает дорогу и к варианту (a)). Решение — начать всё равно с (c):
   ниже риск для старта (не требует ни знания формата встраивания Item в Actor на уровне LevelDB-
   записи на запись, ни ревалидации при каждом обновлении системы `worldofdarkness`), и штатный
   Export Data уже опробован на реальных данных (раздел 1.7) — минимальный путь к рабочему мосту.
   Вариант (a) остаётся следующим шагом, если ручной (c) станет слишком частым/неудобным —
   ограничение раздела 3 (закрывать Foundry на время прогона) с ревизии 4 явно приемлемо для
   пользователя, так что переход к (a) в будущем не заблокирован организационно.
10. **Объём Фазы 1 — только вампиры.** `Mortal`/`Changeling` (свои партиалы в `template.json`,
    свои листы `character_sheet_mortal.md`/`character_sheet_changeling.md` в Sanguine) — отдельная
    будущая задача, не часть текущего плана.
11. **Локации/сцены — вне объёма Фазы 1.** Синхронизация только персонажей; локации/Scene JSON —
    отдельная будущая задача (если понадобится, рекомендация раздела 4 ревизии 1 остаётся в силе:
    однонаправленно Sanguine → Foundry).
