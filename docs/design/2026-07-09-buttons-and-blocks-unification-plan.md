# План унификации кнопок и текстовых/контентных блоков — Sanguine System

Дата: 2026-07-09
Автор: аудит по `DESIGN.md`/`PRODUCT.md` (impeccable, register `product`)

## Вердикт

`DESIGN.md` уже описывает канон (Primary / Secondary-Ghost, радиус 2px везде, карточки
Void Black 3 + Border Blood) — но код разошёлся с ним же самим. Это ровно то нарушение,
которое `PRODUCT.md`/`reference/product.md` называет главным провалом продуктового UI:
*«Consistent affordances across the surface. Same button shape. If the "save" button
looks different in two places, one is wrong.»* Кнопок и текстовых блоков, реализованных
через `cursor:pointer`/`background+border+padding`, в `web/public/styles.css` — **свыше
150 отдельных CSS-классов**, из них десятки — почти побитово идентичные дубли с разными
именами. Ниже — конкретные кластеры (файл/строки), не общие слова.

## 1. Кнопки

### Кластер A — «компактное действие» (26 классов, ~62 места использования)

Идентичный бокс-модель: `padding: 6px 12px; font-family: var(--f-heading); font-size:
var(--fs-2xs); letter-spacing: .1em; text-transform: uppercase; border-radius: 2px;`
Различаются только цветом рамки/текста (по семантике: обычная/edit/danger/gold).

Классы (все — `web/public/styles.css`): `.btn-create-char` (2052), `.btn-img-upload`,
`.cdet-cancel-btn` (2826), `.cdet-edit-btn` (2518), `.cdet-gen-appearance-btn`,
`.cdet-rel-add-btn`, `.cdet-rel-del-btn`, `.cdet-save-btn` (2808), `.cdet-upload-btn`,
`.chd-mod-del-btn`, `.chd-recap-btn`, `.chp-create-btn`, `.chr-delete-btn`, `.city-del-btn`,
`.diary-back`, `.diary-entry-del-btn`, `.hooks-add-btn`, `.mod-fill-add-btn`,
`.mod-gen-scenario-btn`, `.modal-back-btn`, `.module-del-btn`, `.ais-confirm-btn`,
`.ais-restart-btn`, `.modp-back-btn`, `.modp-gen-btn`, `.btn-icon`.

Самые используемые в разметке: `.cdet-edit-btn` (7), `.ais-confirm-btn` (6),
`.modp-gen-btn` (5), `.cdet-save-btn` (5).

**Severity: P1.** Это не 26 разных компонентов — это один компонент, случайно
реализованный 26 раз. Правка hover-состояния или паддинга требует править 26 мест, и уже
сейчас есть дрейф (см. ниже).

### Кластер B — «кнопка листа персонажа» (4 класса-двойника)

`.cdet-sheet-btn` (7208) и `.sheet-btn` (7335) — идентичны по паддингу/радиусу/шрифту,
отличаются только альфа-каналом фона/рамки (`rgba(184,134,11,.12)` vs `.14`,
`.45` vs `.5`) — то есть один и тот же класс с опечаткой в значении, реализованный дважды.
`.modp-canon-btn`, `.modp-npc-dlg-btn` — третий/четвёртый клон того же паттерна.

**Severity: P2.** Визуально почти неотличимы, но это ловушка для будущих правок — поправив
`.sheet-btn`, легко забыть `.cdet-sheet-btn`.

### Кластер C — «главная кнопка» (2 класса-двойника)

`.btn-submit` (1628) и `.chr-modal-btn` (4015) — идентичный паддинг `12px 28px`, тот же
шрифт/радиус — это и есть Primary из `DESIGN.md`, реализованный дважды под разными именами.

**Severity: P1.** Это буквально канонический компонент дизайн-системы, который уже
раздвоился на старте.

### Кластер D — семейство `.modp-*` (страница модуля) живёт на своём радиусе

`.modp-edit-btn` (radius 3px), `.modp-field-regen-btn` (3px), `.modp-npc-del-btn` (3px),
`.modp-promote-btn` (3px), `.modp-cancel-btn` (3px), `.modp-save-btn` (3px, строка 8822),
`.modp-locs-new-btn` (4px) — вся страница модуля использует 3–4px вместо канонических 2px
(`DESIGN.md`, Buttons: `border-radius: 2px` — «почти острые углы»).

**Severity: P2.** Не критично визуально (разница в 1–2px), но это системный дрейф одной
целой страницы от канона, а не случайная опечатка — стоит унифицировать отдельным
проходом именно по `web/public/scripts.js` (рендер `modp-*` вёрстки).

### Настоящий дубль-конфликт: два разных `.danger`

`.chr-modal-btn.danger` (4038, `background: rgba(139,0,0,.15)`, радиус 2px — наследует от
базового `.chr-modal-btn`) и отдельный **`.btn-danger`** (padding `8px 20px`, радиус
**4px**, свой собственный бокс-модель) — два визуально разных «опасных» действия в одном
приложении.

**Severity: P1.** Прямое нарушение `PRODUCT.md`: одна и та же семантика (опасное
действие) выглядит по-разному в разных местах.

### Литеральные rgba вместо токенов

`.cdet-sheet-btn`/`.sheet-btn` используют `rgba(184, 134, 11, 0.12/0.14/0.45/0.5)`
напрямую в правиле вместо переменной golden-тона с альфой. По правилу проекта
(`CLAUDE.md`, «Веб-интерфейс»): «Новые цвета — только через CSS-переменные из `:root`».
Это не новый код, но именно тот тип долга, который сейчас мешает завести единый токен
`--gold-tint-weak/--gold-tint-strong` и потом менять яркость системно в одном месте.

**Severity: P2.**

## 2. Текстовые / контентные блоки

Та же картина, что с кнопками, но по радиусу и паддингу контейнеров (`.modp-info-card`,
`.chd-recap-panel`, `.ais-feat-card` и т.п. — панели с текстом внутри):

| Радиус | Примеры классов |
|---|---|
| 2px (канон) | `.modp-info-card`, `.modp-location`, `.v20-val-report`, `.chd-mod-card`, `.cdet-prompt-box` |
| 3px | `.chd-director-panel`, `.chd-recap-panel`, `.modp-event`, `.thread-form`, `.diary-form`, `.faction-inf-panel` |
| 4px | `.sheet-lib-picker`, `.v20-lib-picker`, `.modp-npc-entry`, `.modp-scenario-block`, `.modp-session` |
| 6px | `.modp-gen-result-inner`, `.toast`, `.ais-feat-card`, `.modp-session-form`, `.ls-section` |
| 8–10px | `.confirm-box` (8px), `.stat-card` (10px) |

**Показательный случай — одна и та же страница модуля, четыре радиуса подряд:**
`.modp-location` = 2px, `.modp-event` = 3px, `.modp-session` = 4px,
`.modp-scenario-block` = 4px — четыре типа блока с текстом на одном экране, каждый со
своим скруглением, без видимой смысловой причины для различия.

**Severity: P1** для страницы модуля (самый плотный экран приложения — GM смотрит на неё
дольше всего за сессию), **P2** для остальных панелей.

`.toast` и `.confirm-box`/`.stat-card` на 6–10px — вероятно осознанное решение (плашки
уведомлений закруглённее панелей, это не противоречит `DESIGN.md`, который явно
разрешает 10px как «редкое исключение для крупных модалок-лайтбоксов» — но `.toast` не
модалка и не лайтбокс, так что даже это стоит сверить).

## 3. Предлагаемая каноническая система

Расширить секцию **Components → Buttons** в `DESIGN.md` до полного набора (сейчас
описаны только Primary/Secondary-Ghost):

1. **Primary** (`.btn-submit`/`.chr-modal-btn` → один класс `.btn-primary`) — как уже
   описано в `DESIGN.md`.
2. **Secondary/Ghost** (`.btn-secondary`) — уже описан, не трогать.
3. **Danger** — один класс, а не два по-разному нарисованных.
4. **Compact action** (новый, документирует кластер A) — `padding: 6px 12px; font-size:
   var(--fs-2xs); border-radius: 2px`, модификаторы `.gold`/`.danger`/`.ghost` для цвета
   рамки/текста — это самый частый тип кнопки в приложении (правка/сохранить/удалить в
   карточках персонажей, модулей, дневников), заслуживает официального места в системе,
   а не 26 клонов.
5. **Icon-only** (документирует `.cdet-delete-btn`, `.modal-close`, `.info-close`) — без
   фона/рамки, только цвет + `font-size`, hover меняет цвет.

Для контентных блоков — один канонический радиус **2px** для всех панелей/карточек с
текстом (уже прописано в `DESIGN.md`, Cards/Containers) — 3/4/6px варианты не codify,
считать их отклонением, которое нужно свести к 2px, кроме явно исключённых
(toast/lightbox, если решим их оставить крупнее).

## 4. Порядок применения (от безопасного к рискованному)

1. **Кластер A и B — DRY без изменения разметки.** Смёржить 26+4 идентичных объявления в
   CSS через один общий селектор/класс-модификатор, **не трогая имена классов в
   `index.html`/`scripts.js`** — визуально ничего не меняется, риск нулевой, правится
   один файл (`styles.css`). Самый быстрый и безопасный первый шаг.
2. **Кластер C (Primary) и Danger-конфликт.** Здесь два по-настоящему разных визуальных
   результата (`.btn-danger` 4px vs `.chr-modal-btn.danger` 2px) — нужно решить, какой
   вид канонический, потом привести второй к нему. Требует одного визуального ревью
   (скриншот before/after), не просто рефактор CSS.
3. **Страница модуля (`.modp-*`, кластер D + контентные блоки 3/4/6px).** Самый большой
   объём (десятки классов в одном месте рендера — `web/public/scripts.js`), но
   изолированный: правки не затрагивают другие страницы. Делать отдельным проходом.
4. **Остальные контентные блоки (панели/формы вне `modp-*`).** Точечно, по мере
   встречаемости — не блокирует остальное.
5. **Токенизация rgba(184,134,11,…) варианты gold-тона** — завести
   `--gold-tint-weak`/`--gold-tint-strong` в `:root`, заменить литералы.

Каждый шаг — отдельная ветка/коммит с `/impeccable audit` до и после (см. правило
`CLAUDE.md` про фронтенд) и визуальной проверкой через `run-sanguine-web` (скриншот
затронутых экранов до/после, не просто «тесты прошли» — здесь никакой node:test не
ловит визуальный регресс).
