### ~~1. Тесты пишут прямо в `cities/paris`, без фикстур~~ — исправлено 2026-07-09 (переоценено вниз)

Копнул глубже перед реализацией: реальная причина, почему каждый прогон
тестов «пачкал» `characters_index.md`/`events.md`/`chronicle.md`, — не
отсутствие изоляции, а то, что в этих файлах был перемешан CRLF/LF (не было
`.gitattributes`-политики), и обычная запись через `fs.writeFile` (в тесте,
в `close_chronicle.js`, в PUT-роуте) молча нормализует их в LF — git видит
это как «изменение», хотя контент тот же. Исправлено разово:
`.gitattributes` (`* text=auto eol=lf`) + `git add --renormalize .`
(коммит `492bcd2`). Проверено: `npm run test:unit` теперь пачкает только
`web/tests/report.html` (штатно перегенерируемый артефакт отчёта, уже
откатывался и раньше).

Полная песочница (temp-копия `cities/`, `CITY_ROOT` из `before()`/`after()`)
была рассмотрена и отклонена: путь к `cities/` независимо хардкодится в 4
местах (`web/lib/db.js`, `web/lib/context_builder.js`,
`web/lib/migrations.js`, `web/routes/modules.js`) — рефакторинг резолвинга
пути в четырёх местах ради выгоды, которую уже даёт `.gitattributes`-фикс,
несоразмерно рискованнее. Понижено в P2 ниже (см. «Что осталось»).

Отдельно: `Date.now()`-слаги в ~20 тестах (строки 1901, 2344, 2461, 2810,
3052, 3147) — теоретический риск коллизии между параллельными прогонами

### 3. `web/public/scripts.js` — 10 408 строк, 224 функции, ноль модульных границ

Все фичи (V20-лист, карточка персонажа, модули, хроники, слухи, города,

персонажа (структурный редактор, library-пикер дисциплин/психики/достоинств/
недостатков, парсинг AI-markdown, оверлей листа, экспорт/импорт Foundry) —

(с 10408). Правки в файлы без бандлера/модулей — глобальная область
видимости общая для всех `<script>`-тегов, поэтому разрезание по границе

порядок объявления имён; проверено отдельно, что ни один top-level
исполняемый вызов (`document.getElementById(...).addEventListener(...)` и
т.п., не внутри функции) внутри вынесенного диапазона не ссылается ни на
что вне `document`, и ни один top-level вызов вне диапазона не ссылается на
имя, определённое только внутри него — только тогда порядок `<script>`-тегов
был бы важен. Проверено вживую (headless Chrome): открыт реальный персонаж,
вкладка «Лист» рендерит полный HTML (65Кб), ошибок в консоли нет.

**Отложено: остальные кластеры (`_lib*` уже частично внутри v20-sheet.js,
`_module*`/сценарии, `_city*`, `_diary*`, `_tour*`, ~6300 оставшихся строк).**
Не тронуты в этом заходе — `scripts.js` намного больше `modules.js` (10408 vs

выносить отдельным проходом с отдельной визуальной проверкой в браузере, а
не одним рывком на весь файл.

## P1 — стоит сделать, не блокирует

### 4. Два разных формата ошибок API в одном приложении — частично исправлено 2026-07-09

Копнув перед правкой обнаружил, что дело не только в форме ответа: 11 мест
(`characters.js`, `generation.js`, `tools.js`, `routes/modules/{lifecycle,
npc,sessions}.js`) в catch-блоке слали клиенту `res.status(500).json({
ok:false, error: e.message })` напрямую, **минуя уже существующий**
`serverError()` из `web/lib/http.js` (импортированный и используемый в этих
же файлах в других местах!) — то есть реальный внутренний текст исключения
(пути, фрагменты стека, тела ошибок от AI-провайдеров) утекал клиенту вместо
общего сообщения с логированием на сервере. Заменил все 11 на `serverError(res, e)`
(коммит `6ab899b`) — это одновременно унифицирует формат (`{error: '...'}`) и
закрывает утечку.

**Не тронуто:** места, где `ok:false` идёт вместе с намеренным 4xx-сообщением
для пользователя (это нормально показывать), и места, форвардящие статус от
внешнего API (OpenAI/OpenRouter) — у них своя семантика, не просто «сервер
упал», трогать их бездумным regex-заменой рискованно без отдельного разбора.
Полная унификация формы ответа (убрать/добавить `ok` везде одинаково) —
более масштабная и менее срочная правка, оставлена как есть.

### 6. Дублирование в frontend: 4 параллельные копии library-паттерна — частично исправлено 2026-07-09

Перед правкой перечитал внимательнее: `_disciplinesCache`/`_psychicsCache`
(плоский массив) и `_meritsCache`/`_flawsCache` (объект по категориям) —
**разные структуры данных**, не одна и та же логика четыре раза. Дисциплины/
психика вдобавок используются и в модалке (клик по названию силы прямо в
листе персонажа), и на отдельной вкладке «Библиотека» — с разными контейнерами
и разной разметкой карточки (клановая принадлежность vs категория/бросок vs
очки достоинства). Слияние в один `renderLibrarySection(kind)`, как
предлагала первая версия плана, потребовало бы подгонять два разных формата
данных под одну форму — это уже не DRY, а натягивание.

Вынес то, что действительно было 5 копипастами одного и того же: делегирование
кликов «список → деталь» / «назад → список» (`e.target.closest('[data-X-slug]')`
+ `e.target.closest('[data-X-back]')`) — теперь один `_bindLibraryClicks()` в
`web/public/v20-sheet.js`, используется во всех 5 местах привязки (модалка
дисциплин + 4 вкладки библиотеки). Сами построители HTML и кэши остались
раздельными — они законно разные. Проверено вживую (headless Chrome): во всех
5 точках список→деталь→назад работает, ошибок в консоли нет.

### 7. Обработка ошибок fetch на клиенте — непоследовательная — частично исправлено 2026-07-09

Перечитал все 10 пустых `catch {}` перед правкой — большинство оказались
**намеренной** деградацией с уже готовым фолбэком прямо в коде, не багами:
`orSettings`/`orModels` (готовый дефолт-объект/`OR_FREE_MODELS_FALLBACK`),
`/api/graph` (готовый `MOCK_GRAPH`), опрос `/api/auth-status` во время
рестарта сервера (сбой там и должен быть тихим — сервер временно не отвечает
по определению), `localStorage` для флага онбординг-тура (само API может
кинуть исключение в приватном режиме — тоже законно тихо). Реально «немой»
силент-фейл был только в двух местах, которые определяют вообще весь список
персонажей/локаций на странице: `ensureCharsLoaded()` (`scripts.js`) и
`ensureLocsLoaded()` (`locations.js`) — сбой сети там неотличим от «в городе
действительно нет персонажей/локаций». Добавил `console.error` в оба (не
полноценный UI-тост — это фоновые загрузчики, вызываемые отовсюду, тост по
каждому вызову был бы шумным; но след в консоли для отладки появился).

Добавил `apiFetch(url, opts)` в `utils.js` — как и рекомендовал план,
opt-in хелпер для нового кода и кода, который правится по другому поводу,
а не отдельный проход по всем 140 вызовам.

## Порядок применения

1. **P0.1 (изоляция тестов)** — разово, но даёт наибольший долгосрочный
   выигрыш: перестаёт требоваться ручная уборка после каждого прогона.
2. **P0.2/P0.3 (разбиение god-файлов)** — делать *постепенно*, начиная с
   самого автономного куска каждого файла (например, `_v20*` во
   frontend, `sessions`-роуты в `modules.js`), а не одним большим рефактором.
3. **P1.4 (унификация формата ошибок)** — можно делать параллельно с
   P0, низкий риск, высокая видимость выгоды для клиентского кода.
4. **P1.5–P1.8** — по мере работы в соответствующих файлах (не отдельным
   проходом) — тесты на dashboard/image-upload, `apiFetch`-хелпер,
   try/catch в CLI.
5. **P2** — оставить как есть, применять только когда затронутый код и так
   меняется по другой причине.
