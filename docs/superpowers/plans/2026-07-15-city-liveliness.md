# План реализации — «живой город» для VtM V20

Дата: 2026-07-15. По анализу «чего не хватает городу» (смертный слой,
география ночи, проактивные НПС, данные для генерации). Развивает фазу D
дорожной карты `2026-07-15-city-page-and-design-roadmap.md`: D расширяется,
добавляются фазы E и F. Порядок: **D1 → D2 → E1 → E2 → F**.

Сквозные правила: TDD (тест → фикс → зелёные), CDP-проверка фронтенда,
уборка тестовых данных, изменение формата пользовательских файлов — только
с миграцией, CSS — токены.

---

## Фаза D1 — новые секции city.md + миграция (M)

**Файлы:** `web/lib/parsers/city.js`, `tools/migrations/2026-07-15-city-sections.js`
(имя по формату `tools/migrations/README.md`), `web/lib/migrations.js`,
`web/tests/all.test.js`.

1. **Расширить `CITY_SECTIONS`** (порядок = порядок секций в city.md):

```js
['political',    'Политический ландшафт'],
['factions',     'Фракции'],
['districts',    'Районы'],                  // NEW: количество и список районов, чей район
['landmarks',    'Значимые места'],          // NEW
['locations',    'Ключевые локации'],
['hunting',      'Охотничьи угодья'],        // NEW: где кормиться, чьё, где запрещено
['edicts',       'Законы домена'],           // NEW: эдикты поверх Традиций
['mortals',      'Смертные институции'],     // NEW: полиция/морг/пресса/криминал и их скомпрометированность
['calendar',     'Календарь города'],        // NEW: фестивали, матчи, годовщины — крючки сцен
['tech',         'Технологии и Маскарад'],   // NEW: камеры, соцсети, риски эпохи
['limits',       'Ограничения генерации'],   // NEW: «Элизиумов ≤2», «в районе ≤4 станций метро»
['leitmotif',    'Лейтмотивы и атмосфера'],
['specifics',    'Специфика ответа'],
['avoid',        'Чего избегать'],
['sources',      'Источники'],
```

   Формы создания/правки города и `buildCityMd`/`parseCityMd` управляются
   этим списком — проверить греп-тестом, что нигде секции не захардкожены
   отдельно (веб-форма new-city, страница/модалка города, `tools/new_city.js`).
2. **Первая реальная миграция**: добавляет отсутствующие `## <заголовок>`
   (с плейсхолдером «*(не заполнено)*») в каждый `cities/*/city.md`;
   идемпотентна (повторный прогон — 0 изменений). Одновременно — по
   плану-2026-07-09 §9 — обернуть `require()` модулей миграций в try/catch
   с внятным логом (сломанная миграция не роняет сервер).
3. **Тесты:** round-trip build/parse с новыми секциями; миграция на
   фикстуре старого city.md (дважды); существующий тест каркаса города.

**Приёмка:** после старта сервера `cities/paris/city.md` и `balmont`
содержат все секции; формы города показывают новые поля; 374+ зелёные.

## Фаза D2 — ограничения и существующие локации в промтах генерации (M)

**Файлы:** `web/lib/context_builder.js`, `web/routes/modules/fill.js`,
`web/routes/locations.js`, `web/tests/all.test.js`.

1. **Хелпер** в `context_builder.js`: `buildCityConstraints(city)` →
   блок текста из секций `limits` + `hunting` + `edicts` + `tech`
   (только заполненные) + компактный список существующих локаций
   (`getAllLocations`: «Название — район», ≤60 строк) с инструкцией:
   «Не создавай новые локации сверх ограничений города; в первую очередь
   используй существующие из списка».
2. **Подмешивание**: в системный промт генерации сценария (`fill.js` —
   все места, где строится baseCtx/промт сцен) и генерации локации
   (`locations.js` `/api/locations/generate`).
3. **Тесты:** AI_MOCK-прогон fill — перехват промта (посмотреть, как мок
   устроен: если промт не сохраняется, добавить в мок-ветку echo промта в
   ответ или лог) — блок «Ограничения города» присутствует, список локаций
   присутствует; города без limits — блок отсутствует (не шумим).

**Приёмка:** заполнить у тестового города «Элизиумов не больше 2» →
mock-генерация сценария содержит блок; unit на `buildCityConstraints`.

## Фаза E1 — часы угроз (S/M)

Механика progress clocks: город меняется между модулями сам.

**Хранение:** секция «⏱️ Часы угроз» в блоке «Состояние мира»
(`events.md`) — таблица `Угроза | Прогресс | Заметка`, формат прогресса
`N/M` (например `2/6`). **Инфраструктура уже есть**: world-state
structured CRUD поддерживает произвольные секции/колонки — ни парсер,
ни эндпойнты менять не нужно.

**Файлы:** `web/public/scripts/archive.js`, `web/lib/context_builder.js`,
`web/routes/modules/lifecycle.js`, `web/tests/all.test.js`.

1. **UI**: в форме «Состояние мира» для секции с заголовком «Часы угроз»
   (матч по «⏱» или тексту) рядом с каждой строкой кнопка «+1 тик»:
   инкремент `N` в `N/M` через существующий PUT строки; при `N == M` —
   подсветка «пробило» (класс на токенах). Кнопка «создать секцию часов»
   не нужна — секция создаётся существующей формой.
2. **Генерация**: `buildCityConstraints` (или отдельный хелпер) добавляет
   в промты сценария блок «Часы угроз: … — учитывай назревающие угрозы»;
   в закрытие модуля (`lifecycle.js` close) — в промт финала инструкция
   «предложи, какие часы продвинулись» (текстом, без автозаписи —
   Рассказчик тикает руками, автозапись канона — вне рамок).
3. **Тесты:** тик через PUT (уже покрыт CRUD-тестами — добавить кейс
   `2/6 → 3/6` и «пробитие» `6/6`), AI_MOCK: промт содержит блок часов.

**Приёмка:** CDP — секция создаётся формой, «+1 тик» двигает прогресс,
6/6 подсвечено; тестовые строки откачены.

## Фаза E2 — мотивация НПС: хочет / боится / рычаг (M)

**Файлы:** `system/schema/card_schema.md`, `system/schema/validate_cards.js`,
`web/lib/db.js` (`EDITABLE_FIELD_MAP`), `web/lib/parsers/character.js`,
`web/public/scripts/char-detail.js`, `web/lib/context_builder.js`,
`system/rules/npcs_city.md` (протокол НПС), `web/tests/all.test.js`.

1. **Контракт**: три опциональных поля карточки —
   `Хочет`, `Боится`, `Рычаг` (по строке каждое). Обновить
   `card_schema.md` (опциональные — миграция НЕ нужна) и линтер
   `validate_cards.js` (знает поля, не требует). Упомянуть в
   `npcs_city.md` и чек-листе нового НПС.
2. **Парсер + правка**: `parseCharacter` читает поля;
   `EDITABLE_FIELD_MAP` + вкладка «Информация» модалки персонажа — три
   новых инпута (та же механика, что статус/локация).
3. **Генерация**: контекст сценария/диалогов НПС (`context_builder` /
   `characters.js` dialogue): если у НПС заполнена мотивация — блок
   «НПС проактивен: хочет …, боится …, рычаг …; в сценах без игроков
   делает шаг к цели».
4. **Генератор НПС** (`fill.js`/npc): новые НПС модуля получают
   мотивацию от AI сразу (поле в промте создания карточки).
5. **Тесты:** parse/PUT round-trip полей; AI_MOCK — блок мотивации в
   промте диалога.

**Приёмка:** CDP — поля редактируются в модалке и сохраняются в карточку;
`validate_cards.js` зелёный на всех городах.

## Фаза F — именник и фактура эпохи (S)

**Файлы:** `cities/<city>/city.md` (секция) ИЛИ `system/library/names/` —
решение: **секция города** `['naming', 'Именник и фактура']` (имена
эпохо- и культуро-зависимы — это данные города, не глобальная библиотека).
Добавляется в D1-список сразу (чтобы не делать вторую миграцию!) —
реализация промтов здесь.

1. Секция: банк имён (мужские/женские/фамилии по слоям общества),
   клановые конвенции, 5–10 строк фактуры (цены, транспорт, сленг эпохи).
2. **Подмешивание**: генерация НПС (`fill.js` npc, `new_npc` веб-путь,
   диалоги) получает блок «Именник города — выбирай имена отсюда, не
   повторяй существующих: <список имён персонажей города>».
3. **Тест:** AI_MOCK — блок именника в промте создания НПС.

**Приёмка:** у города с заполненным именником mock-промт НПС содержит
банк имён и список занятых имён.

## Сводка порядка и рисков

| Фаза | Размер | Ключевой риск |
|------|--------|---------------|
| D1 | M | миграция — первая реальная: прогнать на копии cities/ до пуша |

| F | S | секцию добавить в миграцию D1 заранее |

и полный `npm run test:unit`; после всех фаз — `/impeccable critique`
не нужен (бэкенд/данные), достаточно CDP-проверок форм.
