## Фаза 1 — «Добавить в хронологию мира» из финала модуля (S)

**Цель:** закрытый модуль/хроника попадает в `archive/timeline.md` одним
подтверждением, без ручного перезаполнения формы.

**Точки интеграции (проверены в коде):**
- Финал модуля: `web/routes/modules/fields.js:123` (`PUT .../finale`),
  генерация при закрытии — `web/routes/modules/lifecycle.js:246-258`.
- Запись в хронологию: уже готовый `POST /api/timeline/epoch/:heading/row`
  (`web/routes/archive.js`), парсер `web/lib/parsers/timeline.js`.

**Объём:**
- Фронтенд (`web/public/scripts/modules.js` + `archive.js`): кнопка
  «→ В хронологию мира» рядом с финалом модуля; модалка = существующая
  форма строки эпохи, предзаполненная (год города, тип «Хроника», событие
  из заголовка/первой строки финала, источник — markdown-ссылка на
  `chronicle.md` модуля, резолв как в `_resolveTimelineLink`).
- Выбор эпохи: select по эпохам из `GET /api/timeline/structured`.
- Бэкенд: новых эндпойнтов **нет** — только клиентская компоновка.

**Тесты/приёмка:** CDP-прогон: закрыть тестовый модуль → кнопка → строка
появилась в нужной эпохе `timeline.md`; `git checkout` тестовых следов.
Юнит-тестов не требуется (нет новой серверной логики).

## Фаза 2 — Дашборд «висящих нитей» (S)

**Цель:** нити из `open_threads.md`, которые давно не двигались, видны на
дашборде — забытые сюжеты не теряются.

**Точки интеграции:** `GET /api/threads` (`web/routes/threads.js:37`,
`readThreadsStructured` — уже отдаёт нити из `archive/open_threads.md` и
всех `chronicles/*/open_threads.md`); дашборд — `web/routes/dashboard.js` +
`web/public/scripts/scripts.js` (страница `page-dashboard`).

**Объём:**
- Мера «давности»: у таблицы нитей нет даты последнего изменения → добавить
  в парсер нитей извлечение даты из колонки (если есть) либо fallback:
  `git log -1 --format=%cs -- <file>` на файл нитей (дёшево, файлов ≤ N хроник).
  Решение зафиксировать в спеке фазы.
- Бэкенд: расширить ответ `GET /api/threads` полем `staleDays` (или отдельный
  `GET /api/threads/stale`).
- Фронтенд: блок на дашборде — топ-5 активных нитей по давности, ссылка
  ведёт на страницу нитей (`page-threads`).

**Тесты/приёмка:** unit на вычисление `staleDays` (фикстура-файл);
integration на новый ответ; CDP-проверка блока на дашборде.

## Фаза 3 — Дайс-роллер V20 (M)

**Цель:** броски пулов d10 по правилам V20 прямо в интерфейсе.

**Объём:**
- Чистый фронтенд-модуль `web/public/scripts/dice.js` (новый файл в
  `web/public/scripts/`, подключить в `index.html`): пул, сложность,
  спецправила V20 (10 = успех, 1 съедает успех, botch при 0 успехов и ≥1
  единице, again-правила — что именно включаем, решить в спеке).
- Виджет: плавающая кнопка/панель, доступная с любой страницы (по образцу
  существующих модалок), история последних бросков в сессии.
- Интеграция с листом: на V20-листе (`web/public/scripts/v20-sheet.js`)
  клик по паре «Атрибут + Способность» собирает пул автоматически
  (значения точек уже парсятся листом).
- Опционально (решить в спеке): запись броска в лог сессии через
  существующий `POST /api/log-session` (`web/routes/tools.js:697`).

**Тесты/приёмка:** unit-тесты на чистую функцию броска (детерминированный
RNG через инъекцию) — успехи/единицы/botch/граничные случаи; CDP-проверка
сборки пула с листа. Логика броска — отдельная чистая функция без DOM,
тестируется в `all.test.js` как source-guard/юнит по образцу `utils.js`.

## Фаза 4 — Экран сессии (L)

**Цель:** одна страница для живой игры — без прыжков по вкладкам.

**Состав экрана (всё из существующих API):**
- Выбор активного модуля (`GET /api/chronicles/:slug/modules`).
- Сценарий по сценам (`GET .../scenario`) с навигацией «текущая сцена».
- Быстрые карточки НПС модуля (`GET .../detail` → список НПС; клик —
  существующая модалка `#char-detail-modal`).
- Аудио: кнопка запуска пресета локации текущей сцены
  (`GET /api/audio/presets` фильтр по `locationSlug`; логика запуска уже
  есть в `web/public/scripts/audio-library.js` — вынести в переиспользуемую
  функцию, не копировать).
- Дайс-роллер (фаза 3) встроен панелью.
- Быстрые заметки → по окончании кнопка «Записать сессию» →
  `POST /api/log-session`.

**Объём:** новый `web/public/scripts/session-screen.js` + секция
`page-session` в `index.html` + пункт навигации; новых серверных
эндпойнтов, по предварительной оценке, не требуется (максимум — один
агрегирующий GET, решить в спеке). CSS — на существующих токенах.

**Риск/особенность:** самая большая фаза; в спеке обязательно нарезать на
2–3 независимо сдаваемых куска (каркас+сценарий → НПС+аудио → заметки+лог).

**Тесты/приёмка:** integration на агрегирующий эндпойнт (если появится);
полный CDP-сценарий «открыл экран → листает сцены → запустил пресет →
бросил пул → записал сессию» с уборкой тестовых данных.

## Фаза 5 — Трекер опыта (XP) на V20-листе (M)

> **Закрыта без работ (2026-07-13):** при ревизии перед стартом выяснилось,

> блок «Опыт» (Всего/Потрачено/Остаток), «🎓 Режим опыта» (клик по точке

**Цель:** лог получения/траты опыта с автоповышением точек по ценам V20.

**Объём:**

  причина, остаток). **Меняет формат файлов пользователей → обязательна
  миграция в `tools/migrations/`** (добавление пустой секции старым листам;
  формат — `tools/migrations/README.md`). Это первая реальная миграция —
  по плану-2026-07-09 §9 обернуть `require()` миграций в try/catch.
- Парсер: `web/lib/parsers/` — точечный patch секции по образцу
  `timeline.js`.
- Бэкенд: `GET/POST /api/characters/:slug/xp` (+ `DELETE` строки).
- Цены повышений V20 (атрибут ×4, способность ×2, дисциплина клановая ×5

  автоинкремент точки в листе (правка через существующий механизм
  сохранения листа).
- Фронтенд: блок на вкладке «Лист» (`v20-sheet.js`).

**Тесты/приёмка:** unit на парсер и на таблицу цен; integration на CRUD;
тест миграции (идемпотентность — прогон дважды не дублирует секцию).

## Фаза 6 — История изменений карточки (S)

**Цель:** вкладка «История» в карточке персонажа — просмотр git-истории
файла с диффами (страховка от неудачной AI-перезаписи).

**Объём:**
- Бэкенд: `GET /api/characters/:slug/history` — обёртка над
  `git log --follow --format=... -- <файл карточки>` (+ `?commit=` для
  диффа через `git show`). **Только чтение**, никакой записи/checkout из
  веб-интерфейса. Санитизация: slug валидируется как в остальных роутах
  `characters.js`, команда — через `execFile` с массивом аргументов
  (не строковая интерполяция в shell).
- Фронтенд: вкладка в `#char-detail-modal` (`char-detail.js`), список
  коммитов → раскрытие диффа (рендер моноширинным блоком, подсветка +/-
  двумя CSS-классами на существующих токенах).
- Graceful: если git недоступен/файл вне репо (релизная версия
  пользователя!) — вкладка скрыта, не ошибка. **Проверить поведение в
  ветке `test`**: у пользователя `cities/<город>/` не отслеживается —
  history вернёт пусто, это штатный случай.

**Тесты/приёмка:** integration: история существующего персонажа непуста,
несуществующего — 404, персонаж вне git — пустой список без 500.

## Фаза 7 — Компиляция хроники в «книгу» (M)

**Цель:** закрытая хроника одним читабельным HTML-файлом (печать → PDF
средствами браузера, без серверных PDF-зависимостей).

**Объём:**
- Бэкенд: `GET /api/chronicles/:slug/book` — агрегирует `chronicle.md`
  (спина+финал), события, финалы модулей (`finale.md`), опционально
  дневники участников за период хроники (участники — из карточек модулей).
  Рендер MD→HTML — существующим рендерером проекта (после фикса
  «два mdToHtml» — использовать серверный).
- Отдача: самодостаточный HTML (инлайн-CSS, print-стили `@media print`),
  открывается новой вкладкой, кнопка на странице хроники.
- Зависимостей не добавлять (сейчас 5 runtime — держим).

**Тесты/приёмка:** integration: book закрытой хроники содержит спину,
финал, все модули; незакрытой — тоже работает (без финала); CDP —
открытие вкладки, отсутствие битых относительных ссылок на арт (либо
инлайн base64 для обложки, либо ссылки убрать — решить в спеке).

## Дорожка T — техдолг (фоново, между фазами)

1. **Touch-цели** — один проход `@media (pointer: coarse)`: `.chron-toggle`
   (28px) и родственные мелкие кнопки → ≥44px, десктоп не трогать.
   Делать вместе с любой из фаз 1–2 (обе правят те же экраны).
2. **E2E-smoke `web/tests/e2e/`** — оформить проверенные CDP-сценарии
   (формы timeline/world-state, library-пикер, char-detail) в
   переиспользуемые скрипты по рецепту `.claude/skills/run-sanguine-web`;
   каждая новая фаза добавляет свой сценарий туда, а не в scratchpad.
3. **`styles.css` (11 232 строки)** — разрезать по страницам при следующей
   крупной CSS-правке (вероятно, фаза 4 — она добавляет новую страницу).
   Порядок `<link>` сохраняет каскад; правило `:root`-токенов не меняется.
4. **`apiFetch`** — действующее правило: код, который правится по ходу фаз,
   переводится на `apiFetch` из `utils.js` попутно.

## Сквозные правила для всех фаз

- Каждая фаза: спека (`docs/superpowers/specs/`) → детальный план с кодом
  (`docs/superpowers/plans/`) → выполнение с тестом на каждый шаг → CDP-проверка
  → уборка тестовых данных (`git diff` по `cities/` пуст) → коммит(ы) в `master`.
- Изменение формата пользовательских файлов (фаза 5) — только с миграцией.
- Фронтенд: токены `var(--fs-*)`, цвета из `:root`, touch ≥44px,
  impeccable-прогон до/после (правило CLAUDE.md).
- Релизная сборка: новые файлы дев-окружения — проверить, не нужно ли
  добавить в `tools/release-exclude.json` (e2e-тесты — нужно: `web/tests`
  уже исключён целиком).
