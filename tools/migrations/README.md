# Миграции формата карточек

Когда меняется структура/формат markdown-файлов городов (схема карточки —
`system/schema/card_schema.md`, формат листа персонажа, формат дневника и т.п.) —
**существующие файлы пользователей не правятся вручную**. Вместо этого здесь
добавляется миграция, которая чинит все старые файлы автоматически при
следующем старте сервера (см. `web/lib/migrations.js`, вызывается из
`web/server.js` при каждом `app.listen`).

## Когда нужна миграция

Если правка делает старый формат **невалидным** или **читающимся иначе**:
переименование поля, смена синтаксиса секции, новое обязательное поле без
обратной совместимости. Если старый формат продолжает читаться корректно
(парсер и так понимает оба варианта) — миграция не нужна.

## Формат файла

`NNN_slug.js`, `NNN` — порядковый номер (определяет порядок применения).

```js
module.exports = {
  description: 'короткое описание, что чинит миграция',
  // Должна быть идемпотентной: false на уже мигрированном файле —
  // иначе миграция будет повторно применяться при каждом старте сервера.
  test(text)    { return /- \*\*Старое поле:\*\*/.test(text); },
  migrate(text) { return text.replace(/- \*\*Старое поле:\*\*/, '- **Новое поле:**'); },
};
```

Раннер (`web/lib/migrations.js: runMigrations`) обходит все `cities/<город>/**/*.md`,
прогоняет через `test()` каждую миграцию по порядку номеров и применяет `migrate()`
там, где `test()` вернул `true`. Пишет файл обратно только если что-то изменилось.

## Ручной запуск

```
node tools/run_migrations.js
```

Сервер делает то же самое автоматически при каждом старте — ручной запуск нужен
только для проверки/CI, без поднятия веб-сервера.
