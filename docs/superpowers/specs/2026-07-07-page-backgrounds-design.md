# Тематические фоновые иллюстрации разделов — дизайн

Дата: 2026-07-07

## Проблема

Все 13 верхнеуровневых разделов веб-интерфейса (`web/public/index.html`, `<nav class="nav-menu">`)
выглядят визуально одинаково — один и тот же тёмный фон (`--bg: #07050a`) на дашборде,
персонажах, хрониках, локациях и т.д. Это ослабляет ощущение разных смысловых
пространств приложения.

## Область первой итерации

Только 13 верхнеуровневых разделов, соответствующих пунктам `nav-item[data-page]`:

`dashboard`, `chronicle`, `characters`, `graph`, `factions`, `chronicles-page`,
`modules`, `threads`, `rumors`, `locations`, `library`, `tools`, `search`.

Вложенные виды (`page-module` — детальная карточка модуля, модальные окна,
детальные карточки персонажей/локаций) фон **не** получают — это следующая
итерация, если понадобится.

Светлая тема не затрагивается: сайт сейчас работает только в тёмной палитре
(`:root` не содержит light-override), поэтому иллюстрации и оверлей делаются
только под неё.

## Источник изображений

13 атмосферных готических иллюстраций генерируются через `canvas-design` skill,
по одной на раздел, в едином художественном ключе (тёмная палитра, VTM-сеттинг
Париж 2010). Тематика картинки соответствует смыслу раздела (например: `locations`
— силуэт городской локации, `characters` — абстрактный портретный мотив, `graph`
— переплетение нитей/связей, `library` — книжные полки/гримуар и т.д.) — конкретные
промты подбираются на этапе реализации.

Формат: `.webp`, ориентировочно 1600×1000px, оптимизированы по весу
(целевой ориентир ≤ 200 КБ на файл). Файлы кладутся в
`web/public/img/backgrounds/<page>.webp`, где `<page>` — значение атрибута
`data-page` соответствующего nav-item (например `web/public/img/backgrounds/locations.webp`).

## Архитектура подключения (CSS)

1. Каждая секция `#page-<name>` получает CSS-переменную `--page-bg`, указывающую
   на своё изображение:
   ```css
   #page-dashboard   { --page-bg: url('img/backgrounds/dashboard.webp'); }
   #page-locations   { --page-bg: url('img/backgrounds/locations.webp'); }
   /* ...и так для всех 13 разделов */
   ```
2. Общее правило `.page::before` рисует фон как fixed-слой на весь вьюпорт под
   контентом:
   ```css
   .page {
     position: relative; /* уже может быть задано — проверить существующие стили .page */
   }
   .page::before {
     content: "";
     position: fixed;
     inset: 0;
     z-index: -1;
     background: var(--page-bg) center / cover no-repeat;
     opacity: .35;
     pointer-events: none;
   }
   .page::after {
     content: "";
     position: fixed;
     inset: 0;
     z-index: -1;
     background: linear-gradient(
       to bottom,
       rgba(7, 5, 10, .82),
       rgba(7, 5, 10, .93)
     );
     pointer-events: none;
   }
   ```
   Оверлей (`::after`) обязателен и не настраивается по разделам — единая
   плотность затемнения, чтобы гарантировать читаемость текста на любой
   иллюстрации без ручной подгонки контраста под каждую картинку.
3. Так как в разметке одновременно активна только `.page.active` (остальные
   скрыты), `::before`/`::after` рисуются только у активной секции — лишней
   перерисовки при неактивных разделах нет.
4. Существующие внутренние фоны карточек/панелей (`--bg2`..`--bg4`, `.mod-fill-body`
   и т.д.) не меняются — иллюстрация работает только как фон уровня страницы,
   видимый в просветах между непрозрачными панелями интерфейса.

## Производительность и загрузка

- Изображения не прелоадятся все разом — браузер загружает `background-image`
  лениво по мере активации раздела (обычный CSS-механизм, без доп. JS).
- Дашборд (`dashboard`, активен по умолчанию при загрузке) может получить
  `<link rel="preload" as="image">` в `<head>`, чтобы избежать мигания фона
  при первой отрисовке.
- Ограничение веса файла (~200 КБ) и `cover`-скейлинг делают набор из 13 картинок
  сравнимым по трафику с 1-2 обычными изображениями персонажей.

## Проверка

- Визуальная проверка каждого из 13 разделов в браузере (скилл `run-sanguine-web`):
  фон виден, но не мешает читать текст — приоритет по `code-review`/`impeccable`
  чеклисту CLAUDE.md (контраст, отсутствие литеральных `px`/`#hex` вне токенов
  — оверлей и позиционирование заданы через существующие CSS-переменные там,
  где это уместно).
- Проверка на медленном/узком экране: `background-attachment` через `position: fixed`
  не должен вызывать дёргание при скролле на мобильных (протестировать).
