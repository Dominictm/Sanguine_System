## Вердикт

Проект в целом здоровый: чистый `.gitignore`, нет TODO/FIXME в бэкенде, нет
брошенных spec/plan-документов, зависимости лёгкие (5 runtime), схемы карточек
реально соответствуют коду. Главные риски — **два god-файла** (backend и
frontend) и **тесты, которые пишут прямо в боевые данные `cities/paris`** без
изоляции. Ничего критического/горящего, но откладывать дальше — значит платить
всё больше при каждой следующей фиче.

## P0 — заслуживает отдельного решения до масштабирования

### 1. Тесты пишут прямо в `cities/paris`, без фикстур

`web/tests/all.test.js:57` — `CITY_ROOT = .../cities/paris` (реальные боевые
данные, а не temp-копия). Уборка — вручную по каждому тесту (`DELETE
/api/cities/:slug` и т.п.), нет `before`/`after` на уровне файла. ~20+ тестов
генерируют имена/слаги через `Date.now()` (строки 1901, 2344, 2461, 2810, 3052,
3147) вместо детерминированных фикстур.

**Риск:** упавший на середине прогон оставляет мусор в реальных городских
данных → следующий прогон падает по чужой причине (race через файловую
систему, не память). Это уже частично компенсируется прогоном `git checkout --`
после каждого теста (см. `run-sanguine-web` skill) — но это пластырь, не
решение.

**Рекомендация:** заменить `CITY_ROOT` на temp-директорию с копией
минимального fixture-города, поднимаемую в `before()`/уничтожаемую в `after()`.
Разовая работа на день-два, окупается на каждом последующем прогоне тестов.

### 2. `web/routes/modules.js` — 2543 строки, 30 роутов, 4 домена в одном файле

Хроники/сессии/НПС/сценарии/фракции смешаны в одном модуле — в 3.5 раза больше
следующего по размеру файла (`characters.js`, 1116 строк). `web/lib/parsers.js`
(1249 строк) — аналогично, парсит город/персонажа/хронику/локацию/дневник/
сценарий в одном месте.

**Рекомендация:** разбить `modules.js` на `routes/sessions.js`,
`routes/npcs.js`, `routes/scenarios.js`, `routes/factions.js` (роутер
монтируется под тем же префиксом, изменений в клиенте не требует).
`parsers.js` — разбить по домену аналогично (`parsers/character.js`,
`parsers/chronicle.js` и т.д.), с `parsers/index.js`, реэкспортирующим всё
для обратной совместимости импортов.

### 3. `web/public/scripts.js` — 10 408 строк, 224 функции, ноль модульных границ

Все фичи (V20-лист, карточка персонажа, модули, хроники, слухи, города,
дашборд) в одном файле без `import`/`export`. Проект уже умеет
модуляризировать (`graph.js`, `locations.js`, `rules-v20.js`, `utils.js`
существуют) — просто `scripts.js` исторически не пилился.

**Рекомендация:** выносить по тому же принципу, что уже применён к
`graph.js`/`locations.js` — начать с самого автономного кластера (`_v20*`,
~90 функций) в `v20-sheet.js`, подключаемый `<script>`-тегом рядом с
существующими. Не переход на бандлер — просто разрезание по уже видимым
префиксным кластерам (`_v20*`, `_lib*`, `_module*`/scenario, `_city*`,
`_diary*`, `_tour*`).

## P1 — стоит сделать, не блокирует

### 4. Два разных формата ошибок API в одном приложении

`characters.js`/`generation.js`/`modules.js`/`tools.js` возвращают
`{ ok:false, error }`, а `archive.js`/`chronicles.js`/`cities.js`/`threads.js`/
`library.js` — просто `{ error }` без `ok` вообще. Клиентский код вынужден
понимать, какая форма у конкретного эндпоинта.

**Рекомендация:** унифицировать через `web/lib/http.js` (там уже есть
`serverError` — расширить, чтобы все роуты использовали одну функцию с одним
форматом), one-time рефактор с последующим `grep` на прямые
`res.status(500).json(`, минующие хелпер.

### 5. Неровное покрытие тестами по доменам

`dashboard.js` (335 строк, 4 роута) и `tools.js` (849 строк, 11 роутов) —
почти нулевое упоминание в тестах; `library.js` — только 5 упоминаний
(включая новые `/api/library/merits`/`/flaws`, которые тестами покрыты, но
остальное — нет). Также нет тестов на: загрузку изображений
(`upload-image` роуты), `web/lib/migrations.js`/`tools/run_migrations.js`.

**Рекомендация:** приоритет — `dashboard.js` и image-upload (простые,
быстро закрыть пробел), `tools.js` можно откладывать дольше (CLI, ниже риск
регресса на проде).

### 6. Дублирование в frontend: 4 параллельные копии library-паттерна

`_libRenderDisciplineList/Detail`, `_libRenderPsyList/Detail`,
`_libRenderMeritList/Detail`, `_libRenderFlawList/Detail` — четыре кэша
(`_disciplinesCache`, `_flawsCache`, `_meritsCache`, `_psychicsCache`) с
идентичной структурой list/detail/cache. То же с открытием модалок — 5 пар
`open*Modal`/`close*Modal` без общего хелпера.

**Рекомендация:** один `renderLibrarySection(kind)` вместо четырёх копий —
особенно уместно сейчас, когда только что добавлен четвёртый (merits/flaws) —
пятый (если появится) уже стоит писать сразу через общий хелпер, а не
копировать снова.

### 7. Обработка ошибок fetch на клиенте — непоследовательная

140 `fetch(` вызовов, но только 27 с `.catch(...)`, только 78 проверок `.ok`
(часть из них — вообще не про response, а про `_v20*`-состояние). Есть 10
пустых `catch {}` без обратной связи пользователю (например, строки 534, 988)
и вызовы вообще без try/catch и без `.ok`-проверки (строки 18, 148, 359, 823)
— неуспешный ответ с JSON-телом ошибки будет молча воспринят как успешный.

**Рекомендация:** не переписывать все 140 разом — заводить общий
`async function apiFetch(url, opts)` с единой обработкой не-2xx + единым
UI-фидбеком об ошибке, и переводить на него по мере правки соседнего кода
(а не отдельным большим проходом).

### 8. CLI-инструменты без обработки ошибок

`tools/new_city.js` (0 try/catch), `tools/new_location.js` (0 try/catch),
`tools/run_migrations.js` (0 try/catch) — упадут сырым стектрейсом вместо
понятного сообщения. Хорошая новость: slug-логика **не задублирована** —
`new_location.js`/`new_npc.js` уже импортируют `slugify` из
`web/lib/parsers.js`, `new_city.js` осознанно берёт явный slug-аргумент
(документированное решение, не долг).

**Рекомендация:** обернуть точку входа каждого CLI-скрипта в один
try/catch с `console.error` + `process.exit(1)` — 10-минутная правка на файл.

## P2 — можно отложить на потом

### 9. `tools/migrations/` — механизм есть, но ни разу не использован

`web/lib/migrations.js` (77 строк) чистый и с задокументированным контрактом
(`test`/`migrate`, обязана быть идемпотентной), но `tools/migrations/`
содержит только `README.md` — ни одной реальной миграции ещё не было. Плюс
плохая ошибка не изолирована: `require()` сбойного модуля миграции уронит
сервер целиком при старте (миграции гонятся синхронно в `runMigrations`).

**Рекомендация:** не трогать, пока не появится первая реальная миграция
формата — но при первом её добавлении сразу обернуть `require()` каждого
модуля в try/catch с понятным логом (какой файл, какая ошибка), не давая
сломанной миграции обрушить весь сервер.

### 10. `.github/workflows/release-test.yml` — брендовое совпадение якорей

`release-overrides.json` — всего 2 записи (`start.bat`, `CLAUDE.md`),

хорошо. Пока только 2 записи — не проблема, но список хрупкости растёт
линейно с числом override-записей.

**Рекомендация:** ничего делать сейчас не нужно — просто держать в уме при
следующей правке `start.bat`/`CLAUDE.md`, что anchor может не совпасть

## Порядок применения

1. **P0.1 (изоляция тестов)** — разово, но даёт наибольший долгосрочный
   выигрыш: перестаёт требоваться ручная уборка после каждого прогона.
2. **P0.2/P0.3 (разбиение god-файлов)** — делать *постепенно*, начиная с
   самого автономного куска каждого файла (например, `_v20*` во
   frontend, `sessions`-роуты в `modules.js`), а не одним большим рефактором.
3. **P1.4 (унификация формата ошибок)** — можно делать параллельно с
   P0, низкий риск, высокая видимость выгоды для клиентского кода.
4. **P1.5–P1.8** — по мере работы в соответствующих файлах (не отдельным
   проходом) — тесты на dashboard/image-upload, `apiFetch`-хелпер,
   try/catch в CLI.
5. **P2** — оставить как есть, применять только когда затронутый код и так
   меняется по другой причине.
