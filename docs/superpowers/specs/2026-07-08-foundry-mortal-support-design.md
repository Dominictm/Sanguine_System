# Дизайн: Foundry sync — поддержка Mortal + массовый экспорт (Фаза 2)

> Продолжение [`2026-07-08-foundry-integration-design.md`](2026-07-08-foundry-integration-design.md)
> (Фаза 1 — только вампиры, уже в `master`). Эта ревизия расширяет двусторонний Sanguine ⇄ Foundry
> экспорт/импорт на линейку «смертные» и добавляет массовый (по выбору) экспорт нескольких
> персонажей в Foundry одним ZIP-архивом. **Changeling — вне рамок**, отдельный будущий цикл
> (требует новых полей в модели листа: Glamour/Banality/Сферы/Искусства, которых сейчас в
> `_v20Empty()` вообще нет).

---

## 1. Входные данные

Пользователь предоставил два реальных Foundry Export Data (`worldofdarkness` v7.1.5, мир `parizh-20`):
`смертный.json` (`Actor.type: "Mortal"`, NPC-охранник) и `фея.json` (`Actor.type: "Changeling"`).
Оба читаются с явным mojibake кириллицы (та же кодировочная особенность, что была отмечена в
Фазе 1 — не блокирует разбор структуры, но подтверждает необходимость explicit UTF-8 при парсинге
реальных файлов пользователя).

**Подтверждённые находки по `Actor.type: "Mortal"`:**

- `system` **не содержит** ключей `clan`/`sect`/`generation`/`sire`/`bloodline`/`weakness`/`custom`
  вообще — это не пустые поля вампирской схемы, этих ключей нет в шаблоне Mortal в принципе.
- `system.settings` (флаги `has*`) в образце: `haswillpower: true`, всё остальное
  (`haspath`, `hasvirtue`, `hasbloodpool`, `hasrage`, `hasglamour`, …) — `false`. Это конкретная
  NPC-заглушка («Охранник»), не обязательный эталон: Sanguine **ведёт** для смертного Человечность
  (Добродетели → Человечность) и Волю (`system/rules/character_sheet_mortal.md`, шаг 5), поэтому при
  экспорте **из Sanguine** маппер сам проставляет `haspath: true`, `hasvirtue: true`,
  `haswillpower: true` для Mortal — собственный дефолт под то, что Sanguine реально хранит, а не
  копирование конкретного NPC-образца.
- Новый тип предмета: `Item.type: "Trait"`, `system.type: "wod.types.othertraits"` (8 штук в
  образце — Dead-Eyes, Sense Lifeline, Manifest и т.п., все `value: 0, max: 0`, `isrollable: false`
  — текстовые «особенности» без дотов). Соответствует уже существующему в модели листа Sanguine
  полю `otherTraits` (свободные текстовые черты, страница 2 листа) — этого типа раньше в мапперах
  не было вовсе (ни для вампиров).
- Кастомные способности (`wod.types.{talent|skill|knowledge}secondability`) и Фон
  (`wod.types.background`) уже реализованы в мапперах **лайнэдж-агностично** (не завязаны на
  проверку `"vampire"`) — заработают для Mortal без единой правки.
- Оружие/броня (`Ranged Weapon`, `Melee Weapon`, `Armor`) — Sanguine их не моделирует ни для одной
  линейки; вне рамок и здесь.
- `powers.hasnumina: false`, `Power`-предметов в образце нет — у нас **нет реального образца схемы**
  для Нумины/Граней (`sheetData.psychicPowers`). В отличие от Фона (где была убедительная аналогия
  `wod.types.<kind>` с достоинствами/недостатками/дисциплинами), здесь гадать не на чем.

## 2. Решения (обсуждены и приняты с пользователем)

| # | Вопрос | Решение |
|---|---|---|
| 1 | Разбивать Mortal/Changeling на два цикла или проектировать сразу оба? | Разбить — **сейчас только Mortal**, Changeling отдельным будущим циклом (нужны новые поля Glamour/Banality/Realms/Arts в модели листа) |
| 2 | Что делать с `psychicPowers` без образца схемы? | **Пропустить** — не экспортировать/не импортировать в этой фазе; явный комментарий-маркер в коде, TODO на будущую проверку с реальным психиком-смертным |
| 3 | Какой экспорт получает массовый выбор карточек? | **Экспорт в Foundry** (не общий raw-md бэкап-экспорт `GET /api/export/characters` — тот не трогаем) |
| 4 | Формат вывода массового экспорта? | **ZIP с отдельными `foundry_<slug>.json`** — Foundry Import Data принимает по одному Actor за раз на листе конкретного актёра, так что бандл должен оставаться набором отдельных файлов, не единым массивом |
| 5 | Библиотека для сборки ZIP? | **Свой мини-модуль без внешних зависимостей** (`web/lib/zip.js`, на встроенном `zlib`) — в проекте сейчас нет ни одной ZIP-библиотеки, ничего специфичного ZIP от нас не требует полноценного стороннего пакета |

## 3. Изменения в мапперах (`web/lib/foundry-export.js` / `foundry-import.js`)

### 3.1 `mapCharacterToFoundryActor` (экспорт Sanguine → Foundry)

- `type`: `'Vampire'` или `'Mortal'` по `char.lineage`, вместо жёсткого `'Vampire'`.
- Клан/секта/поколение/кровь/`custom` — вычисляются и пишутся в `system` **только при
  `char.lineage === 'vampire'`**; для Mortal эти ключи в объекте `system` не создаются вообще
  (не пишутся как пустые строки — их не должно быть, по образцу раздела 1).
- `settings.has*`: два пресета по линейке.
  - Vampire — без изменений (текущий блок).
  - Mortal (новый): `haswillpower: true, haspath: true, hasvirtue: true, hasbloodpool: false`,
    остальные `has*` (`hasrage`, `hasgnosis`, `hasglamour`, `hasbanality`, `hasnightmare`,
    `hasconviction`, `hasfaith`, `hastorment`, `hasessence`, `hascorpus`, `haspathos`, `hasangst`,
    `hasvitality`, `hasspite`, `hasbalance`, `hassekhem`, `hasquintessence`) — `false`.
- Новая функция `_otherTraitItems(otherTraits)` — по образцу `_traitItem()`, но без `group`/уровня
  способности: `Item.type: "Trait"`, `system.type: "wod.types.othertraits"`, `value` из
  `sheetData.otherTraits[i].val`, `max: 0` (сами по себе бездотовые в исходном образце, но
  Sanguine-сторона может хранить дот — не обрезаем).
- `psychicPowers` — не экспортируются (нет вызова, нет соответствующей функции-маппера).
- `_disciplineItems`/`_backgroundItems`/`_meritFlawItems`/`customTraits`/`_mapAbilities`/
  `_healthDamage` — без изменений, уже лайнэдж-агностичны.

### 3.2 `mapFoundryActorToSheetData` (импорт Foundry → Sanguine)

- Обратное чтение `wod.types.othertraits` → `sheetData.otherTraits` (по аналогии с уже
  существующим чтением `wod.types.discipline`/`wod.types.background`).
- Чтение `sys.clan`/`sys.sect`/`sys.generation` уже условно (`sys.clan ? ... : null`) — для Mortal
  просто не сработает без единой правки; уточняем это явным комментарием в коде, чтобы не
  выглядело как забытый случай.
- `psychicPowers` не читаются (соответствует пропуску при экспорте).

## 4. Роут и UI (одиночный экспорт/импорт — Mortal)

- `web/routes/characters.js`, `GET /api/characters/:slug/export-foundry`: гейт
  `char.lineage !== 'vampire'` → `!['vampire', 'mortal'].includes(char.lineage)`; текст ошибки
  обновляется соответственно.
- `POST /api/characters/:slug/import-foundry` — гейта по линейке уже нет, изменений не требует
  (клан/секта/поколение для Mortal и так придут `null`/пустыми — клиентский тост уже условный,
  проверено по коду).
- `web/public/scripts.js`, тулбар листа персонажа (~строка 7437): кнопки «Экспорт/Импорт в Foundry»
  сейчас показываются по условию `isVamp ? ... : ''`; добавляется `isMortal = (m.lineage || '') ===
  'mortals'`, условие становится `(isVamp || isMortal) ? ... : ''`.

## 5. Массовый экспорт в Foundry (новая функциональность)

### 5.1 Поток взаимодействия

1. Пользователь на странице персонажей нажимает новую кнопку **«Экспорт в Foundry»** —
   включается режим выбора.
2. В режиме выбора карточки `.char-card` **только поддерживаемых линеек** (вампиры, смертные)
   становятся кликабельными для выбора (визуальная рамка/отметка при выборе); карточки прочих
   линеек (феи и т.д.) визуально приглушены и не реагируют на клик по выбору — Changeling пока не
   поддержан маппером, предлагать его в списке было бы обманом ожиданий.
3. Повторный клик по той же кнопке (теперь показывает счётчик, например «Экспортировать (3)»)
   подтверждает выбор и скачивает ZIP.
4. Отдельная кнопка **«Отмена»** выходит из режима выбора без экспорта, сбрасывает выбор,
   восстанавливает обычное поведение кликов по карточкам (открытие модалки персонажа).

### 5.2 API

Новый роут `POST /api/characters/export-foundry-bulk` (`web/routes/characters.js`):

- Тело запроса: `{ slugs: string[] }`.
- Для каждого слага: найти персонажа (как в одиночном `export-foundry`), проверить
  `['vampire','mortal'].includes(char.lineage)` — при несовпадении тихо пропустить (UI и так не
  даёт выбрать неподдерживаемые карточки, но сервер не доверяет клиенту вслепую), прочитать
  `<slug>-sheet.json`, вызвать существующий `mapCharacterToFoundryActor`.
- Собрать ZIP через `web/lib/zip.js`: один файл `foundry_<slug>.json` на персонажа.
- Ответ: `Content-Type: application/zip`, `Content-Disposition: attachment; filename="foundry_export_<город>_<YYYY-MM-DD>.zip"`.
- Пустой список `slugs` или все слаги пропущены (неверная линейка/не найдены) → `400` с понятным
  сообщением, а не пустой ZIP.

### 5.3 `web/lib/zip.js` — ZIP без внешних зависимостей

- Экспортирует `createZip(files) → Buffer`, где `files: { name: string, data: Buffer|string }[]`.
- Формат: **ZIP store (без сжатия)** для простоты и надёжности (JSON-файлы персонажей небольшие,
  а генерация корректного deflate + CRC32 + local header + central directory вручную — это уже
  весь необходимый объём работы; ужимать эти файлы deflate — не обязательное усложнение при первой
  реализации). Реализация: local file header + data по каждому файлу, затем central directory,
  затем end-of-central-directory record — стандартная минимальная ZIP-структура, читается
  Windows/`unzip`/Foundry без стороннего инструмента.
- CRC32 считается вручную (таблица на 256 записей, ~15 строк) — единственная нетривиальная часть,
  без неё большинство распаковщиков валидацию не пройдут.

### 5.4 UI/CSS

- `web/public/scripts.js`: состояние `_foundryBulkMode` (bool) + `_foundryBulkSelected` (Set слагов).
- Рендер списка персонажей — в режиме выбора добавляется класс на поддерживаемые карточки
  (например `.char-card.fdry-selectable`) и обработчик клика, переключающий
  `.char-card.fdry-selected`; неподдерживаемые карточки получают `.char-card.fdry-disabled`
  (приглушение через CSS-переменную непрозрачности, не литеральный цвет).
- Кнопки `#btn-export-foundry-bulk` (переключает режим / подтверждает) и `#btn-export-foundry-cancel`
  (виден только в режиме выбора) — рядом с существующей `#btn-export-chars`.
- Новые CSS-классы — только через существующие токены (`var(--fs-*)`, переменные `:root` для
  цвета/непрозрачности), без литеральных `px`/`#hex`, по правилу проекта для фронтенда.
- Перед правками `styles.css`/`scripts.js`/`index.html` — `/code-review`; после — повторный
  impeccable-прогон дизайн-системы (см. `CLAUDE.md`, раздел «Веб-интерфейс»).

## 6. Тестирование

- `web/tests/all.test.js`: round-trip юнит-тест на очищенном (не mojibake) фикстур-объекте по
  мотивам `смертный.json` — кастомная способность (`Trait`/`talentsecondability`), Фон, одно
  достоинство/недостаток (сверка с библиотекой), один `othertraits`-предмет, здоровье, воля,
  Человечность/Путь. Проверяется: (а) экспорт Sanguine→Foundry не пишет клан/секту/поколение для
  Mortal, (б) реимпорт того же Actor даёт эквивалентный `sheetData`.
- Тест на `_otherTraitItems`/обратное чтение — отдельно, минимальный сценарий (одна черта туда и
  обратно).
- Тест на `web/lib/zip.js` — `createZip` от 2–3 файлов, распаковать обратно средствами Node
  (`zlib`/ручной парсер ZIP в тесте, или проверка через системную `unzip`, если доступна в CI —
  предпочтительно чистый JS-парсер в самом тесте, без внешних тулов) и сверить содержимое байт-в-байт.
- Тест на `POST /api/characters/export-foundry-bulk`: happy path (2 персонажа разных
  поддерживаемых линеек → ZIP с 2 файлами), пустой список → 400, список из одних неподдерживаемых
  линеек → 400, смешанный список (часть поддержана, часть нет) → ZIP только с поддержанными.

## 7. Явно вне рамок этой фазы

- **Changeling** — отдельный будущий цикл (нужны новые поля модели листа: Glamour, Banality,
  Realms, Arts — их пока просто некуда мапить).
- **Нумина/`psychicPowers`** — нет образца реальной Foundry-схемы, откладывается до появления
  психика-смертного в реальном экспорте.
- **Оружие/броня** (`Ranged Weapon`, `Melee Weapon`, `Armor`) — Sanguine не моделирует ни для одной
  линейки, не только для Mortal.
- **`era`/`variant`/`dicesetting`** в `system.settings` — существующий пробел, общий с вампирским
  маппером Фазы 1, не новый для Mortal, не расширяется в этой фазе.
- **Общий raw-md экспорт персонажей** (`GET /api/export/characters`) — не трогается, отдельная
  независимая функциональность.
