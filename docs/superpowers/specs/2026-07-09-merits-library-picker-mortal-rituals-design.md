# Дизайн: Достоинства/Недостатки из справочника + Ритуалы на листе смертных

Дата: 2026-07-09

## Проблема

**Достоинства и недостатки.** Раздел листа V20 (`sheetData.meritsFlaws`) — одно
свободнотекстовое поле, по строке на пункт («Внушительный тип (1 очко)»). В
отличие от Дисциплин/Фона, тут нет ни кнопки «+ Из справочника», ни
структурированных строк с удалением по одной. При этом справочник уже
существует (`system/library/merits/*.json`, `system/library/flaws/*.json`,
JSON с полями `name`/`points`/`category`/`description`) и уже используется
сервером — `web/lib/foundry-merits.js` строит по нему индекс для сверки текста
при экспорте в Foundry. Предыдущий спек по библиотечным пикерам
(`2026-07-08-sheet-library-picker-design.md`) явно выносил Достоинства/
Недостатки за рамки первой итерации именно потому, что справочника тогда не
было — теперь он есть, разрыв можно закрыть.

Дополнительно: `_v20ParseMd` (парсер ИИ-сгенерированного markdown-листа)
сейчас **не разбирает** таблицу «Достоинства и недостатки (Merits & Flaws)»
вообще — хотя ИИ её генерирует (`character_sheet_v20.md`, секция
«⚠️ Слабости, изъяны, деранжементы»). После генерации листа поле остаётся
пустым, всё приходится набирать в текстовое поле руками.

**Ритуалы у смертных.** Колонка «Ритуалы» на листе (`m.rituals`) рендерится
только при `isVamp` (канонически — Таумturgy/Некромантия, вампирская
механика). Пользователь хочет видеть её и на листе смертных — хоумрул-гибкость
для конкретной хроники (например, смертный оккультист/культист, изучивший
ритуалы). Модель листа (`_v20Empty()`) уже содержит поле `rituals` для всех
линеек — ограничение чисто в рендере.

## Часть 1 — Достоинства/Недостатки: модель данных

`sheetData.meritsFlaws`: строка → массив `[{name, points, kind}]`, где
`kind: 'merit' | 'flaw'`. `points` — не точки-доты 0–5 (в библиотеке
встречаются costs до 7), а обычное число, вводится текстовым/числовым полем,
не кликабельными точками.

**Обратная совместимость.** Существующие сохранённые листы хранят это поле
строкой. Отдельная миграция (`tools/migrations/`) не подходит — та система
работает с markdown-карточками городов через regex-тесты на текст, а не с
JSON sheet-data персонажей. Вместо неё — тот же приём, что уже применён в этой
кодовой базе для дисциплин/фона/othertraits: авто-нормализация на клиенте в
`_v20Normalize()`.

Новая функция `_v20MigrateMeritsFlawsString(text)`:
- Разбивает текст по строкам, для каждой непустой строки:
  - Извлекает имя (убирает маркер списка `- `, скобки, суффикс очков —
    та же логика, что в `foundry-merits.js: _extractName()`, портированная на
    клиент).
  - Ищет имя (нормализованное) в объединённом клиентском индексе
    достоинств+недостатков (см. Часть 2 — тот же справочник, что грузит
    пикер). Если нашли — берёт `points`/`kind` оттуда.
  - Если не нашли (кастомная строка) — `points` парсится из `(N)` в конце
    строки (0, если нет числа), `kind` — **по умолчанию `'merit'`** (старый
    формат не хранил тип вообще; это не регресс, а более полная информация,
    чем было — просто ставится безопасный дефолт, если ничего лучше нет; GM
    видит переключатель Достоинство/Недостаток на каждой строке и может
    поправить руками при следующем открытии листа).
- Вызывается в `_v20Normalize()`, если `typeof m.meritsFlaws === 'string'`.

`_v20PadMeritsFlaws`/`_v20TrimTrailingEmpty`-подобная обрезка хвостовых пустых
строк — без защищённой базовой длины (баз. длина 0 — как у дисциплин/фона
после недавнего фикса «динамических строк»).

## Часть 2 — UI

Одна кнопка **«+ Из справочника»** — объединённый список достоинств и
недостатков всех категорий (умственные/социальные/физические/
сверхъестественные). Загружается через два новых эндпоинта:

```
GET /api/library/merits  → [{name, points, category, description}, ...]  (все категории слиты)
GET /api/library/flaws   → [{name, points, category, description}, ...]
```

(`web/routes/library.js`, переиспользуют уже существующие
`getAllMerits()`/`getAllFlaws()` из `web/lib/merits-loader.js`/
`flaws-loader.js` — те же функции, которыми пользуется `foundry-merits.js`
для построения индекса сверки.)

Клиент грузит оба эндпоинта параллельно, помечает каждую запись `kind` по
источнику, сортирует одним списком. Строка в пикере: `Название · Достоинство
· Умственное · 2 очк.` (по аналогии с хинтом «N уровней» у дисциплин).
Клик — заполняет первую пустую строку модели (как `_v20AddLibraryItem`),
включая `kind`.

Кнопка **«+ Добавить»** — пустая редактируемая строка вручную (`{name: '',
points: 0, kind: 'merit'}` по умолчанию).

Рендер строки: текстовое поле «Название», переключатель Достоинство/
Недостаток (`<select>` на 2 опции — простое и однозначное решение, не
радиокнопки на каждой строке), числовое поле «Очки», кнопка «✕ Удалить»
(`.v20-row-remove-btn`, без защищённой базовой длины).

## Часть 3 — Парсинг из ИИ-листа

В `_v20ParseMd`: строки таблицы «Достоинства и недостатки» — 3 колонки
(Название/Тип/Очки) внутри секции с заголовком, содержащим «слабост» (совпадает
с `sec` для «⚠️ Слабости, изъяны, деранжементы» — единственная таблица с 3
колонками в этой секции, отличается от прозы «Клановая слабость:»/«Изъян
(Flaw):»/списка деранжементов, которые не являются markdown-таблицами).
`kind` определяется по содержимому колонки «Тип»: если встречается
«недостат» (регистронезависимо) — `'flaw'`, иначе `'merit'`.

## Часть 4 — Foundry-синхронизация

`web/lib/foundry-export.js`: `mapCharacterToFoundryActor` разветвляет по
`typeof s.meritsFlaws`:
- **Массив** (новый формат) — каждая запись экспортируется как `Item` типа
  `Feature` напрямую (`system.type: wod.types.${kind}`, `level: points`), без
  сверки с библиотекой — `kind`/`points` уже известны из самой модели листа.
  Заменяет `_meritFlawItems(matchedMeritsFlaws)`.
- **Строка** (старый формат, ещё ни разу не открытый в браузере после
  обновления) — прежнее поведение через `matchMeritsFlaws()` (сверка текста
  с библиотекой, несовпавшее — в `system.notes`). `matchMeritsFlaws()` в
  `foundry-merits.js` не удаляется, остаётся веткой для обратной
  совместимости.

`web/lib/foundry-import.js`: `mapFoundryActorToSheetData` возвращает готовый
массив `[{name, points, kind}]` из Feature-Item'ов
(`wod.types.merit`/`wod.types.flaw`) вместо склеенной строки. Текст из
`system.notes` (несовпавшие вручную вписанные заметки) добавляется отдельными
записями массива с `points` из `(N)`-суффикса, если есть, иначе `0`, и
`kind: 'merit'` по умолчанию (тот же безопасный дефолт, что в Части 1, —
`system.notes` в принципе не хранит тип).

## Часть 5 — Ритуалы на листе смертных

`web/public/scripts.js`, `_v20RenderSheet`: условие рендера колонки «Ритуалы»
(`isVamp ? ... : '<div class="v20-col"></div>'`) меняется на
`(isVamp || isMortal) ? ... : ...`. Данных/API это не касается — `m.rituals`
уже общее поле модели для всех линеек.

`system/rules/character_sheet_mortal.md` — короткая пометка (1–2 строки), что
Ритуалы доступны как опциональный хоумрул для конкретной хроники (не канон
базового V20 для смертных), чтобы не выглядело как забытая правка правил при
следующем чтении документа.

## Тестирование

- Юнит-тесты на `_v20MigrateMeritsFlawsString`-эквивалент — не применимо
  напрямую (клиентский JS без node:test окружения для DOM-зависимого кода,
  как и остальной `_v20*` код листа) — проверяется вручную через
  `run-sanguine-web` (загрузить старого персонажа со строковым meritsFlaws,
  убедиться, что после открытия листа сформировался массив с корректными
  points/kind для известных библиотеке достоинств).
- `web/tests/all.test.js`: юнит-тесты на оба пути `mapCharacterToFoundryActor`
  (`meritsFlaws` как массив и как строка), на `mapFoundryActorToSheetData`
  (Feature merit/flaw → массив, notes → доп. записи), на новые роуты
  `GET /api/library/merits` / `GET /api/library/flaws`.
- Ручная проверка в браузере (`run-sanguine-web`): пикер открывает
  объединённый список, клик заполняет пустую строку с правильным `kind`,
  «+ Добавить» создаёт пустую редактируемую строку, «✕» удаляет; колонка
  «Ритуалы» появляется на листе смертного.

## Вне рамок

- Валидация «этого достоинства не должно быть у этой линейки/клана» — как и
  у дисциплин (см. `2026-07-08-sheet-library-picker-design.md`), выбор не
  ограничивается и не помечается предупреждением.
- Изменение формата `character_sheet_v20.md`/`character_sheet_mortal.md`
  таблицы «Достоинства и недостатки» — таблица остаётся как есть, меняется
  только то, что Sanguine теперь умеет её парсить.
- Массовая перезапись существующих `-sheet.json` файлов на диске в новый
  формат — миграция происходит лениво, при следующем открытии листа в
  браузере (см. Часть 1), не батч-скриптом по всем городам.
