# Добавление дисциплин/нумина из справочника в лист персонажа — дизайн

Дата: 2026-07-08

## Проблема

Редактор листа персонажа (`web/public/scripts.js`, `_renderSheetEditor` /
`openSheetOverlay(ctx, 'edit')`) сейчас умеет только менять количество точек у
**уже существующих** строк таблиц «Дисциплины» и (в теории) «Нумина / Грани» —
добавить дисциплину или психическую способность, которых ИИ не сгенерировал
при создании/перегенерации листа, можно только руками правя markdown-файл
листа вне структурированного редактора. При этом справочники по дисциплинам
(`system/library/disciplines/`) и психическим способностям
(`system/library/psychics/`) уже существуют, уже отдаются как
`GET /api/library/disciplines` / `GET /api/library/psychics`, и уже
используются сервером при генерации листа — но не доступны пользователю
напрямую в UI редактирования.

## Область первой итерации

Только **Дисциплины** (клановые силы вампиров) и **Нумина / Грани**
(психические способности смертных-экстрасенсов). «Достоинства и недостатки»
(Merits & Flaws) — вне охвата: для них в проекте ещё нет справочника с
очками/описаниями (только таблица клановых слабостей внутри
`character_sheet_v20.md`), создание такого справочника — отдельная будущая
задача.

Без клановой валидации: выбор дисциплины не из клановых для персонажа не
блокируется и не помечается предупреждением (внеклановые дисциплины бывают —
через Достоинства/особые случаи).

## UI

В модалке `sheet-overlay` (режим `edit`), под таблицей каждой группы,
чья секция/подсекция — «Дисциплины» или «Нумина / Грани», появляется кнопка
**«+ Добавить из справочника»**. По клику разворачивается инлайн-комбобокс:
текстовый фильтр по названию + список отфильтрованных записей библиотеки,
каждая — с кратким описанием (для дисциплин: `note` дисциплины или `clans`;
для нумина: `category`). Выбор одной записи:

1. Добавляет новую строку в конец таблицы этой группы с 1 очком (`●○○○○`).
2. Закрывает комбобокс.
3. Новая строка сразу доступна для правки очков через существующий
   `_dotControl`/`_onSheetDotClick`.

У каждой строки таблицы (и существующей, и только что добавленной) появляется
кнопка **✕** — удаляет строку из группы. И кнопка добавления, и кнопка
удаления — не меньше 44px по высоте на `pointer: coarse` (CLAUDE.md,
touch-цели).

Список библиотеки грузится один раз при открытии модалки в режиме `edit`
(если в листе есть хотя бы одна подходящая группа) и кэшируется в
`_sheetEditState` на время открытия модалки.

## Данные и сборка markdown

### Парсинг (`_parseSheetForEdit`)

Для каждой группы дополнительно фиксируются `firstLineIdx` и `lastLineIdx` —
индексы первой и последней строки таблицы в исходном массиве `lines` (это уже
можно получить из `group.rows[0].lineIdx` / `group.rows[at(-1)].lineIdx` в
момент разбора, до каких-либо изменений).

Регэксп `_SHEET_EDIT_SECTIONS` расширяется: `нумина` добавляется в
перечисление (`/атрибут|способност|преимуществ|дисциплин|предыстор|добродетел|нумина|(?<!производные )характеристик/i`),
поэтому раздел «Нумина / Грани» смертных-экстрасенсов вообще попадает в
структурированный редактор (сейчас не попадает никак).

### Добавление строки

Новая строка — объект той же формы, что парсер создаёт для существующих строк
таблицы, но без `lineIdx` (она не существует в исходном файле):

```js
{
  name: '<название из справочника>',
  value: 1,
  lineIdx: null,
  cells: ['<название>', '', ''],
  rating: { value: 1, dotsIdx: 1, numIdx: 2, combinedIdx: -1 },
}
```

Форма `cells`/`rating` подобрана так, что `_rebuildSheetRow(cells, rating, v)`
(уже существующая функция) рендерит её в ту же 3-колоночную разметку
(`| Название | ●○○○○ | 1 |`), что и обычные строки — отдельного рендер-пути
для новых строк не требуется.

### Удаление строки

Просто убирается из `group.rows` и `parsed.editable` (сплайсом по ссылке на
объект строки, не по индексу — индексы в `editable` не обязаны совпадать с
позицией в `group.rows`).

### Сборка (`_buildEditedSheet`)

Вместо точечной правки `parsed.lines[r.lineIdx]` для каждой строки —
пересборка диапазона строк на группу:

```js
function _buildEditedSheet(parsed) {
  const lines = parsed.lines.slice();
  const ordered = parsed.groups.slice().sort((a, b) => b.firstLineIdx - a.firstLineIdx);
  for (const g of ordered) {
    const rowLines = g.rows.map(r => _rebuildSheetRow(r.cells, r.rating, r.value));
    lines.splice(g.firstLineIdx, g.lastLineIdx - g.firstLineIdx + 1, ...rowLines);
  }
  return lines.join('\n');
}
```

Группы обрабатываются от конца файла к началу (`sort` по убыванию
`firstLineIdx`), поэтому `firstLineIdx`/`lastLineIdx` групп, ещё не
обработанных (расположенных выше по файлу), не сдвигаются предыдущими
сплайсами. Один и тот же код обслуживает: правку очков (без add/remove —
результат идентичен старому поведению), добавление (splice вставляет
дополнительную строку) и удаление (splice убирает строку без замены).

Если из группы удалены все строки (`g.rows.length === 0`), splice просто
убирает исходный диапазон, не вставляя ничего — под заголовком таблицы
останется пустая таблица (шапка + разделитель без строк данных), это
допустимо.

## Известные ограничения (осознанно не решаются в этой итерации)

- Если в листе персонажа секция «Дисциплины»/«Нумина» присутствует как
  заголовок, но не содержит вообще ни одной строки данных — парсер не создаёт
  для неё объект группы (текущее поведение `_parseSheetForEdit`, не
  меняется), и кнопке «+ Добавить» негде появиться. Для вампиров это не
  проблема (клановые дисциплины есть у любого персонажа по протоколу
  создания, Шаг 4). Для смертных без псионики раздел «Нумина» по протоколу
  вообще не генерируется — отсутствие кнопки корректно отражает то, что
  добавлять пока некуда.
- Достоинства/недостатки — не в этой итерации (см. «Область первой
  итерации»).
