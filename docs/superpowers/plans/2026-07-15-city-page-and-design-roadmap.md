# План — веб-дизайн: анимации, город как страница, ограничения генерации

Дата: 2026-07-15. Запрос пользователя: (1) больше анимаций; (2) создание
города — отдельным элементом навигации; (3) описание города — со страницы,
а не из модалки (по образцу модуля); (4) расширить поля описания города
(районы, значимые места, ограничения — «не больше 2 элизиумов», «в районе
не более 4 станций метро») и использовать ограничения при генерации
сценариев, чтобы не плодить лишние локации.

> Каждая фаза перед реализацией проходит свой цикл «спека → детальный план»
> (конвенция проекта). Здесь — объём, точки интеграции, порядок.

## Фаза B — «Новый город» в навигации (S)

**Сейчас:** форма создания города живёт вкладкой `new-city` в «Инструментах»;
из сайдбара к ней ведёт маленький «+» у переключателя городов
(`#btn-city-add`, `data-nav="tools" data-tab="new-city"`).

**Объём:**
- Новый пункт навигации «🏙️ Новый город» в группе «Архив» (или отдельной
  строкой под переключателем городов — решить в спеке) → новая секция
  `#page-city-new`; форма переносится из вкладки Инструментов целиком
  (разметка + обработчики в `city.js`), вкладка `new-city` в Инструментах
  удаляется, «+» у переключателя перенацеливается на новую страницу.
- Роутинг: ветка в `navigate()`; тур (`tour.js`) — проверить шаги,
  ссылающиеся на вкладку new-city.

**Приёмка:** CDP — создание тестового города со страницы работает
(и удаляется после), старые точки входа ведут на новую страницу,
373+ тестов зелёные (тест создания города через API не затрагивается).

## Фаза C — город как страница (M)

**Сейчас:** карточка города в «Инструментах» открывает модалку
`#city-detail-modal` (`openCityDetail`, `city.js:291`) с полями
display/год/описание и 7 секциями `CITY_SECTIONS` (textarea-редактирование,
`PUT /api/cities/:slug`).

**Объём (по образцу страницы модуля `page-module`/`openModulePage`):**
- Секция `#page-city` + `openCityPage(slug)`: шапка (титул Cinzel, год,
  бейдж активного города, кнопка «Сделать активным»), блоки-секции как
  карточки с инлайн-редактированием (та же механика textarea → PUT, но
  по-блочно, как правка блоков сценария на странице модуля).
- Карточки городов в Инструментах и клик по названию домена в сайдбаре
  ведут на страницу; модалка `#city-detail-modal` удаляется (разметка,
  обработчики, CSS).
- Breadcrumb «← Инструменты» / возврат.
- Серверных изменений нет (GET/PUT `/api/cities/:slug` уже есть).

**Приёмка:** CDP — открытие страницы города, правка секции с сохранением
и перечтением; удалённая модалка не упоминается нигде (grep).

## Фаза D — новые поля города + ограничения генерации (M)

**Сейчас:** `CITY_SECTIONS` (`web/lib/parsers/city.js:10`) — 7 секций;
`buildCityMd`/`parseCityMd` полностью управляются этим списком; веб-формы
(создание и редактирование) рендерят секции из него же. Контекст генерации
(`web/lib/context_builder.js`) подмешивает `city.md` целиком.

**Объём:**
1. **Новые секции** в `CITY_SECTIONS`:
   - `districts` — «Районы» (количество и список районов с краткой сутью);
   - `landmarks` — «Значимые места» (знаковые точки города);
   - `limits` — «Ограничения генерации» (свободные строки-правила:
     «Элизиумов в городе не больше 2», «В каждом районе не более 4 станций
     метро», «Не создавать новые ночные клубы — использовать существующие»).
   Формы создания/правки города подхватывают секции из списка автоматически
   (проверить, что обе действительно итерируют CITY_SECTIONS, а не хардкод).
2. **Миграция** `tools/migrations/` (первая реальная): добавить пустые
   секции в существующие `city.md` пользователей; идемпотентна; при её
   добавлении — обернуть require миграций в try/catch
   (план-2026-07-09 §9). Прогоняется автоматически при старте сервера.
3. **Генерация.** Ограничения уже попадут в промты неявно (city.md
   целиком), но этого мало — сделать явно:
   - `web/routes/modules/fill.js` (генерация сценария) и
     `web/routes/locations.js` (генерация локации): в системный промт
     добавляется блок «Ограничения города» из секции `limits` +
     инструкция: «Не выдумывай новые локации сверх ограничений; в первую
     очередь переиспользуй существующие локации города (список прилагается)»;
   - список существующих локаций города уже доступен (`getAllLocations`) —
     передавать компактным перечнем (название + район), чтобы модель
     ссылалась на них, а не плодила дубли.
4. **Тесты:** unit на parse/build с новыми секциями (round-trip уже
   покрыт — расширить фикстуру), тест миграции (дважды — без дублей),
   integration: AI_MOCK-прогон fill — промт содержит блок ограничений
   (в AI_MOCK промт можно перехватить/проверить логикой мока — уточнить
   в спеке, как мок отдаёт использованный промт).

**Приёмка:** город со заполненными «Ограничениями» → в сгенерированном
сценарии (mock) блок ограничений присутствует в промте; старые города
после миграции валидны; CDP — новые поля редактируются на странице города.

## Фаза A — motion pass (M, финальная)

Принципы: продукт-регистр — «motion conveys state, not decoration»,
150–250мс, `transform`/`opacity`, токены `--dur-*`/`--ease`, обязательный
`prefers-reduced-motion` фолбэк (уже есть паттерн у `.dice-verdict`).

**Конкретные точки (не «везде», а по состояниям):**
1. **Переход между страницами**: `.page.active` — лёгкий fade+rise
   (opacity 0→1, translateY 6px→0, ~180мс) — сейчас страницы переключаются
   мгновенным display.
2. **Card-reveal на остальных сетках**: паттерн `card-reveal` (есть у
   карточек персонажей) распространить на сетки локаций, хроник, городов,
   аудио-карточек — ступенчатый delay по индексу, максимум ~8 ступеней.
3. **Аккордеоны** (сцены сессии, события «Подробнее», финал): плавное
   раскрытие через `grid-template-rows: 0fr→1fr` (без анимации height).
4. **Дайс-роллер**: каскад кубиков-чипов (stagger ~30мс) перед вердиктом —
   у вердикта reveal уже есть.
5. **Тосты**: вход/выход через transform+opacity, если ещё нет.
6. **Нав-группы**: hover-подсветка пунктов — оставить как есть (whisper),
   без новых эффектов; активный пункт — мягкое появление кровяной полосы.
7. **Инлайн-формы** (правка строк хронологии): появление формы fade+rise.

**Чего не делать:** оркестрованных последовательностей при загрузке страниц,
bounce/elastic, анимаций layout-свойств, декоративного движения в покое
(анти-паттерны продукт-регистра и DESIGN.md).

**Приёмка:** CDP-замеры (transition/animation в computed style на точках
1–7), `prefers-reduced-motion: reduce` эмуляцией — все анимации гаснут;
визуальный скрин-чек; полный прогон тестов.
