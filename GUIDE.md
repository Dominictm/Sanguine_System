## 1. Что это и как устроено

Система состоит из двух частей:

- **Данные** — обычные Markdown-файлы в `cities/<город>/` (персонажи, хроники, модули, локации) и правила в `system/rules/`. Их можно править руками или через веб.
- **Веб-интерфейс** (`web/`) — локальный сервер для просмотра, генерации и ведения игры. Ничего не уходит в облако, кроме запросов к выбранному ИИ-провайдеру.

> Принцип: **источник истины — файлы**. Веб только удобно их показывает и редактирует. После любой генерации полезно посмотреть `git diff` перед коммитом.

## 2. Установка и первый запуск

```bash
# 1. Зависимости веба
cd web && npm install

# 2. Запуск сервера
node server.js            # → http://localhost:3000
```

На Windows можно запускать `web/start.bat`, останавливать — `web/stop.bat`.

Открой `http://localhost:3000`. Слева вверху — **селектор города** (пока пустой; создадим город в шаге 4).

> **Важно про перезапуск.** Изменения во фронтенде (`public/*`) подхватываются обновлением страницы (F5). Изменения в `server.js` требуют перезапуска сервера (`stop.bat` → `start.bat`).

## 3. Настройка ИИ-провайдеров

Генерация (проза, модули, реплики и т.д.) работает через **любой** из провайдеров. Просмотр и ручная правка ИИ не требуют.

Открой **Инструменты → 🤖 Модели AI**. Слева — авторизация провайдеров, справа — назначение их по функциям.

### 3.1. Авторизация (хотя бы один провайдер)

- **🌐 OpenRouter** — вставь `OPENROUTER_API_KEY`, подтверди. Есть бесплатные модели.
- **🤖 OpenAI (GPT)** — вставь `OPENAI_API_KEY`. Только по ключу (входа «через аккаунт ChatGPT» для API нет).
- **🧠 Claude (Anthropic)** — на выбор:
  - **Вход через Claude Code** (без ключа): нажми «🔓 Войти через Claude Code» → откроется страница Claude → авторизуйся → скопируй показанный код → вставь в поле → «✓ Подтвердить код».
  - **API-ключ** `ANTHROPIC_API_KEY` — как альтернатива.
  - «🔄 Обновить статус» — перечитать состояние входа (например, после `/login` в терминале Claude Code).

Сохранённые ключи пишутся в `web/.env`. После сохранения сервер перезапускается сам (или перезапусти вручную).

### 3.2. Назначение провайдеров по функциям

В блоке **«⚡ Назначение провайдеров»** у каждой функции — переключатель **OpenRouter / GPT / Claude** и список моделей:

| Функция | Назначение |
|---|---|
| 👁 Внешность по арту | Vision-анализ изображений (нужна мультимодальная модель) |
| 📍 Генерация локаций | Карточки мест при наполнении модуля |
| 🪄 Генерация прозы | Дневники и финалы сессии |
| 💬 Генерация фраз | Реплики НПС в характере |

Выбери провайдера/модель → «✓ Применить». Можно смешивать: например, проза — Claude, реплики — бесплатный OpenRouter.

> Если выбранный Claude упрётся в лимит (429), система сама сделает повтор и при необходимости переключится на резерв (GPT → OpenRouter), пометив это в результате.

## 4. Создание города

Город — самодостаточная папка `cities/<slug>/`. Создаётся утилитой:

```bash
node tools/new_city.js london "Лондон" 2010
```

`london` — ASCII-слаг (имя папки), `"Лондон"` — отображаемое имя, `2010` — год сеттинга.

Затем:

1. Опиши сеттинг в `cities/london/city.md` — политика, фракции, ключевые локации, лейтмотивы, чего избегать, источники. Это «конституция» города; ИИ сверяется с ней при генерации.
2. Обнови страницу веба и выбери город в **селекторе слева вверху** — весь интерфейс переключится на него (`?city=london`).

## 5. Персонажи

### Создание через веб
**Инструменты → Новый НПС**. Для вампира поля: Имя (обяз.), Клан (обяз.), Секта (обяз.), Поколение, Год рождения, Год обращения, Сир, Биография, Внешность, опционально загрузка арта. Карточка создаётся по правилам `system/rules/npcs_city.md`, имя добавляется в индекс города.

### Создание вручную
Карточка — `cities/<город>/characters/<линейка>/<slug>/<slug>.md`. Линейки: `vampires`, `fairies`, `mortals`, `werewolves`, `mages`, `hunters`.
Обязательные поля контракта (`system/schema/card_schema.md`): H1 (эмодзи + имя), `Слаг` (= имя папки), `Родной город`, `Линейка WoD`, `Статус`, секция `## 🖼️ Изображения`.

### Просмотр и правка
**Персонажи** → двойной клик по карточке. В модалке:
- **Описание** — поля, внешность, генерация промта изображения, генерация **реплик НПС**.
- **Лист** — V20-лист персонажа.
- **Дневники** — записи по месяцам + генерация прозы.
- **Отношения** — редактирование связей по записям (имя из списка существующих или своё; тип связи стандартный или произвольный).

Арт кладётся в `art/` персонажа; «Внешность по арту» (Vision) опишет внешность по изображениям.

## 6. Хроники и модули

**Хроника** — крупная сюжетная арка (`cities/<город>/chronicles/<хроника>/`). **Модуль** — отдельный сценарий внутри хроники (`modules/<модуль>/`).

Жизненный цикл модуля — три фазы (правила: `system/rules/module_rules.md`).

### Фаза A — создание и генерация

1. **Модули** (или **Инструменты → Модуль**) → «+ Модуль». Заполни базовое: название, **время действия** (важно для проверки таймлайна), участников, и **концепцию** (`## 💡 Концепция` — основная идея). Создаётся стаб-документ и папка.
2. Открой страницу модуля → «🪄 Сгенерировать». По концепции ИИ построит:
   - `scenario.md` — предпосылки, локации, НПС, завязка, сцены, кульминация, финалы, открытые нити, колорит города;
   - карточки **локаций** (переиспользует существующие однотипные, не плодит дубли);
   - карточки **эпизодических (модульных) НПС** в `npc/<slug>/`.
3. При генерации ИИ:
   - **проверяет таймлайн** — если на дату модуля персонаж не мог участвовать или имел иной статус (ещё не на должности, в торпоре, не обращён, уже уничтожен), добавляет предупреждение «⚠️ Проверка таймлайна»;
   - **учитывает связи** между персонажами;
   - даёт **имя каждому НПС** (без безымянных функций).

**Вкладки страницы модуля:** Краткое · Сценарий · Сессии · События · НПС · Локации · Нити.
На вкладке **НПС**: имя — ссылка на карточку; под каждым НПС — окно «💬 Реплика НПС» для генерации ответа в характере прямо во время игры.

## 7. Ведение игры: журнал сессий (Фаза B)

Не каждый модуль проходится за один вечер. Вкладка **«🎲 Сессии»** на странице модуля — журнал по сессиям.

1. **+ Запись сессии**: дата, статус (🟡 В процессе / 🟢 Завершён).
2. **Сцены** выбираются из выпадающего меню (берутся из сценария; уже сыгранные исключаются). Постоянный пункт **«События вне сценария»** — для отступлений от сценария (игроки не всегда идут по плану). За одну сессию можно отметить несколько сцен.
3. Опиши, что произошло: события, решения котерии, последствия. Сохрани.
4. Записи можно **редактировать** (✏️).

Кнопка «🔍 Проверить канон» сверит текст записи с установленными фактами (см. §12).

## 8. Закрытие модуля (Фаза C)

Когда модуль доигран — на странице модуля «🔒 Закрыть модуль». По правилам Фазы C ИИ:
- сгенерирует литературный **финал** (`finale.md`) по тому, что реально произошло в игре (а не только по плану);
- перенесёт **каноничное событие** в `events.md` хроники;
- пометит модуль закрытым.

Рядом — «Удалить модуль»: вычищает записи модуля из хроник, ссылки в дневниках персонажей и неканоничных НПС, созданных только в этом модуле.

> Закрытие пишет канон — действие сложно отменить. Проверь `git diff` перед коммитом.

## 9. Генерация прозы: дневники и финалы

После логирования сессии создаются заготовки дневников и финалов с меткой «⏳ ОЖИДАЕТ ГЕНЕРАЦИИ».

**Инструменты → 📓 Сессия** (или страница персонажа → Дневники) → кнопка генерации прозы. ИИ напишет записи строго по `diary_rules.md`: глубокий POV, клановый стиль автора, Маскарад через метафоры, факты — только из событий хроники.

Провайдер берётся из «Генерация прозы»:
- **Claude** (вход через Claude Code) — пишет агентно, правит файлы напрямую (нужен установленный Claude Code).
- **OpenRouter / GPT** — одним запросом возвращает текст, сервер раскладывает по файлам.

## 10. ИИ-инструменты в сцене

| Инструмент | Где | Как использовать |
|---|---|---|
| **💬 Реплики НПС** | карточка персонажа (Описание) · вкладка «НПС» модуля | Опиши ситуацию → получи реплики в характере (поле **Голос** + клановый стиль) |
| **🔍 Проверка канона** | панель целостности · форма сессии | Вставь текст сцены → ИИ подсветит противоречия с фактами |
| **«Ранее в хронике…»** | для игроков | ИИ-рекап последних событий |
| **Внешность по арту** | карточка персонажа | Загрузи арт в `art/` → опиши внешность (Vision) |
| **Промт изображения** | карточка персонажа | Сгенерируй positive/negative-промт по секции «Внешность» |

Реплики НПС работают и для модульных (неканоничных) НПС — карточка ищется в `npc/` модуля.

## 11. Связи, локации, фракции, слухи

- **Связи** — D3.js-граф отношений города. Типы связей (Семья, Сир/Чайлд, Союзник, Враг, Преданность, Нейтральный, Знакомый, Тайная связь) с цветами. Редактирование связей — в модалке персонажа, вкладка «Отношения».
- **Локации** — карточки мест (`cities/<город>/locations/...`): сенсорная палитра, VtM-контекст, крючки, промт.
- **Фракции** — карта политических сил (`archive/political_state.md`).
- **Слухи** — Элизиума / Грёзы (d20).

## 12. Проверки целостности и канона

**Панель** (дашборд) содержит панель целостности:
- битые внутренние ссылки и прочие проверки структуры;
- блок **«🔍 Проверка непротиворечий канону»** — вставь текст сцены/события, ИИ сверит с установленными фактами (статусы НПС, даты, кто где/когда, «мёртвый» персонаж в сцене) и подсветит противоречия.

> Качество поиска противоречий зависит от модели: датовую арифметику («в 2009 он ещё не Шериф») надёжно ловят сильные модели (Claude/GPT), бесплатные — хуже. Для канон-чека выбирай модель посильнее.

Дополнительно:
- **Инструменты → Проверка** — валидации в вебе.
- CLI: `node system/schema/validate_cards.js` (линтер карточек), `tools/validate_links.ps1` (ссылки, Windows).

## 13. Несколько городов

Каждый город независим. Создавай через `new_city.js`, переключай селектором слева.
Персонаж принадлежит одному `Родной город`. Если он появляется в другом городе — оформи это ссылкой в `cities/<другой>/archive/visitors.md` (один источник истины, без копий карточки).

## 14. Утилиты командной строки

| Задача | Команда |
|---|---|
| Создать город | `node tools/new_city.js <slug> "<Название>" [год]` |
| Запустить веб | `cd web && node server.js` |
| Пересобрать индекс событий города | `node tools/build_city_events.js <город>` |
| Линтер карточек | `node system/schema/validate_cards.js` |

## 15. Частые вопросы и проблемы

**Кнопки генерации не работают / ошибка «Нет источника для генерации».**
Не настроен ни один ИИ-провайдер. Зайди в **Инструменты → 🤖 Модели AI** и авторизуй OpenRouter, OpenAI или Claude.

**Ошибка 429 (rate limit).**
Лимит запросов у провайдера. Система сделает повтор и при необходимости переключится на резерв. Если упорно — подожди минуту или выбери для этой функции OpenRouter (бесплатные модели).

**Claude пишет «токен истёк».**
Зайди в Модели AI → «🔓 Войти через Claude Code» заново, либо выполни `/login` в Claude Code и нажми «🔄 Обновить статус». При наличии refresh-токена обновление происходит автоматически.

**Внешность по арту не получается.**
Для функции «👁 Внешность по арту» нужна **мультимодальная** модель (Gemma Vision, Llama Vision, Qwen VL, gpt-4o, Claude). Выбери её в назначении провайдеров.

**Изменения в `server.js` не применяются.**
Сервер не перезагружает себя сам при ручной правке кода. Перезапусти: `web/stop.bat` → `web/start.bat`.

**Город не виден в селекторе.**
Убедись, что папка `cities/<slug>/` создана через `new_city.js` и есть `city.md`; обнови страницу.

**Куда смотреть перед коммитом.**
ИИ-генерация и закрытие модулей пишут файлы. Всегда просматривай `git diff` — это твой контроль качества.
