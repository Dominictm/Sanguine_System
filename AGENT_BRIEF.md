# AGENT_BRIEF.md — инженерный бриф для делегированных задач

Это **не** `CLAUDE.md`. `CLAUDE.md` в корне — основной, живой файл проекта: он задаёт роль
Рассказчика, активный город, литературный стиль, протокол НПС и т.д., и реально
управляет основной сессией. Этот файл — урезанный инженерный контекст для агента
(сабагента через Agent-тул, отдельный worktree, внешний/облачный агент), которому
для задачи нужен **только** дев-контракт проекта, без нарративной части.

Используется в паре с [`AGENT_TASK_TEMPLATE.md`](AGENT_TASK_TEMPLATE.md) — тот
описывает конкретную задачу, этот файл — статический контекст проекта.

---

## Стек и окружение

- **Runtime:** Node.js (нативные `fetch`/`WebSocket`/`node --test` — нужен
  относительно свежий Node; в разработке использовался Node 22+), Windows
  (PowerShell 5.1 / Git Bash)
- **Веб-сервер:** `web/server.js` (Express), порт 3000. Запускается через
  `web/wrapper.js` (`npm start` в `web/`) — супервизор, перезапускающий
  `server.js` по exit-коду 75; есть `POST /api/restart` для применения правок
  кода к уже запущенному серверу без падения терминала
- **Фронтенд:** SPA в `web/public/` (`index.html`/`scripts.js`/`styles.css`/
  `rules-v20.js`) — ванильный JS, никаких бандлеров и npm-зависимостей на клиент
- **ИИ-провайдеры:** Anthropic / OpenRouter / OpenAI — маршрутизация в
  `web/server.js` (`makeGenerationClient`); `AI_MOCK=1` — детерминированный
  мок-провайдер для тестов, без сетевых вызовов
- **Данные:** Markdown-файлы в `cities/`, `system/` — никакой БД; общая
  пар­синг-библиотека `web/lib/parsers.js` (single source of truth и для
  сервера, и для CLI-тулов в `tools/`)

## Структура проекта (ключевые точки)

```
cities/<город>/                    — данные города (создаётся через tools/new_city.js)
  city.md                          — сеттинг (описание + секции: политика, фракции,
                                      локации, лейтмотивы, специфика, чего избегать, источники)
  archive/                         — political_state.md, events.md (агрегат — см. ниже),
                                      timeline.md, characters_index.md, visitors.md
  characters/<линейка>/<slug>/     — карточки персонажей (<slug>.md, -sheet.md/.json, art/, journal/)
  locations/district_NN/<район>/   — карточки локаций
  chronicles/<хроника>/            — chronicle.md, events.md, open_threads.md, modules/
system/rules/                      — глобальные правила (npcs_city, module_rules, diary_rules,
                                      literary_style, character_sheet_v20/mortal/changeling …)
system/schema/card_schema.md       — контракт полей карточки персонажа
system/schema/validate_cards.js    — линтер карточек
web/server.js                      — API и ИИ-маршрутизация
web/lib/parsers.js                 — парсеры карточек/локаций/связей/дневников (общий для веба и tools/)
web/.env                           — ключи провайдеров (НЕ в git)
web/tests/                         — node:test (`all.test.js` юнит/интеграция, `e2e.test.js` сквозной цикл)
tools/                             — CLI-утилиты (new_city, new_npc, new_location, build_city_events,
                                      validate_links, migrate_char, close_chronicle, search, backup …)
```

Активный город для конкретной задачи указывается в заполненном
`AGENT_TASK_TEMPLATE.md`, а не здесь — этот файл городонезависимый.

## Команды

| Задача | Команда |
|---|---|
| Запустить веб (супервизор, рекомендуется) | `cd web && npm start` |
| Запустить веб напрямую (без супервизора) | `cd web && node server.js` |
| Применить правки кода к уже запущенному серверу | `curl -X POST http://localhost:3000/api/restart` |
| Тесты (юнит + e2e) | `cd web && npm test` |
| Создать город | `node tools/new_city.js <slug> "<Название>" <год:3–4 цифры>` (год обязателен) |
| Создать персонажа (CLI) | `node tools/new_npc.js <город> <линейка> "<Имя>" <Мужской\|Женский> [...]` |
| Пересобрать индекс событий города | `node tools/build_city_events.js <город>` |
| Линтер карточек | `node system/schema/validate_cards.js` (`--strict` в e2e-тестах) |
| Проверить внутренние ссылки | `tools/validate_links.ps1` (Windows) |

После любых изменений карточек персонажей — прогнать линтер. После правок
в `web/` — `npm test` в `web/`; перед коммитом убрать шум из
`web/tests/report.html` (`git checkout -- web/tests/report.html` — файл
перегенерируется со свежим таймстампом при каждом прогоне, не несёт diff-сигнала).

Для проверки фронтенда в браузере — скилл `.claude/skills/run-sanguine-web/`
(headless Chrome через сырой CDP, без Puppeteer/Playwright в проекте).

## Что НИКОГДА трогать без явного указания в задаче

- `web/.env` — никогда не читать, не изменять, не выводить содержимое
- `cities/*/archive/events.md` — агрегат; пересобирается только через `build_city_events.js`
- `system/schema/card_schema.md` — контракт карточки; изменения ломают парсер и линтер
- `web/lib/parsers.js` — менять только если задача явно касается парсинга (это общий
  модуль для сервера и CLI — drift между ними уже был источником багов)
- Данные конкретных городов (`cities/<город>/`), если задача не про этот город явно
- Ветку/воркतри `test` (релизная заготовка) — не трогать без отдельного указания

## Соглашения по карточкам персонажей

- Имя персонажа — **кириллицей** в карточке (H1); папка — ASCII-слаг
  (`web/lib/parsers.js: slugify`)
- Обязательные поля: H1 (эмодзи + имя), `Слаг`, `Родной город`, `Линейка WoD`,
  `Статус`, секция `## 🖼️ Изображения`
- Полный контракт — `system/schema/card_schema.md`
- После создания/правки карточки — `node system/schema/validate_cards.js`

## Соглашения по коду

- Стиль существующего кода в файле — эталон; не менять стиль без задачи на рефакторинг
- Фронтенд (`web/public/`) — ванильный JS, никаких npm-зависимостей на клиент
- API-маршруты в `web/server.js` — city-aware: `reqCity(req)` читает `?city=<slug>`
  (по умолчанию — первый существующий город), нужен везде, где код трогает данные города
- Не добавлять `console.log` в продакшн-код без явной пометки `// TODO: remove`
- При правках `web/public/*` — см. правило impeccable/web-design-guidelines из
  основного `CLAUDE.md` (токены `--fs-*`/`:root`, контраст, touch-таргеты ≥44px)

## Работа с Markdown-контентом

- Канонные факты WoD (имена, даты) — только по источникам; при неоднозначности
  писать `// нужна сверка` / `⚠️ Требуется уточнение`, не выдумывать
- Структуру папок города не менять руками — только через `tools/new_city.js`
- Персонажи-гости из других городов — ссылка в `cities/<город>/archive/visitors.md`
  принимающего города, не копировать карточку
